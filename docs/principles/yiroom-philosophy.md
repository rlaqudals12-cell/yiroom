# 이룸 철학 (Yiroom Philosophy)

> **이 문서는 이룸의 모든 설계 판단의 최상위 기준이다.**
> 다른 원리 문서, ADR, 스펙과 충돌 시 이 문서가 우선한다.

---

## 1. 궁극의 명제

```
"이룸은 사용자가 자기 자신을 통합적으로 이해하여,
 스스로 판단할 수 있게 돕는 앱이다."
```

### 1.1 핵심 구분

| 이룸이 하는 것               | 이룸이 하지 않는 것      |
| ---------------------------- | ------------------------ |
| 분석 결과 간 연결을 보여준다 | 사용자 대신 결정한다     |
| "왜"를 설명한다              | 무조건적 지시를 내린다   |
| 판단 능력을 키운다           | 영구적 의존을 만든다     |
| 자립을 목표로 한다           | DAU 극대화를 목표로 한다 |

### 1.2 슬로건과의 연결

```
"온전한 나는?" (Know yourself, wholly.)

온전한 = 파편이 아닌 통합 (피부+컬러+체형+영양+운동이 연결된)
나 = 외부 기준이 아닌 자기 자신
? = 탐구 과정 (답이 아닌 여정)
```

### 1.3 성공의 정의

```
이룸의 성공 = 사용자가 이룸 없이도 자신에게 맞는 선택을 할 수 있는 상태

역설: 성공할수록 사용자는 앱을 덜 필요로 한다.
해결: 새로운 분석 모듈과 깊이가 지속적 가치를 제공한다.
```

## 2. 3-Layer 가치 모델

```
Layer 3: 자기 이해 (Insight)          <- 궁극의 가치 (부산물)
         "나는 이런 사람이구나"
              | 축적
Layer 2: 판단력 성장 (Competence)      <- 핵심 가치 (전달 가능)
         "이래서 이게 나한테 맞는 거구나"
              | 반복
Layer 1: 즉각 해결 (Utility)           <- 진입 가치 (후크)
         "내 피부 타입이 뭐지?"
```

### 2.1 각 Layer의 역할

**Layer 1 — 즉각 해결 (Utility)**

- 사용자가 앱을 처음 쓰는 이유
- "퍼스널컬러가 뭐지?", "내 피부 타입은?"
- 단일 분석 결과 제공
- 경쟁 앱도 할 수 있는 영역

**Layer 2 — 판단력 성장 (Competence)**

- 이룸만의 핵심 가치
- "봄 웜톤이라서 코랄 립이 어울리는 거구나"
- 분석 간 교차 인사이트 + "왜" 설명
- 반복 노출로 사용자가 직접 판단 가능해짐

**Layer 3 — 자기 이해 (Insight)**

- 궁극의 가치이지만 직접 전달 불가
- Layer 2의 축적으로 자연스럽게 도달
- "나는 웜톤에 건성이고 상체 발달형이니까..."
- 통합적 자기 인식 = 슬로건 "온전한 나는?"의 실현

### 2.2 설계 원칙

```
1. Layer 1으로 진입시키되, Layer 2에서 가치를 느끼게 한다.
2. Layer 3을 직접 목표로 삼지 않는다 (부산물).
3. 모든 기능은 "이 기능이 어떤 Layer에 기여하는가?"로 검증한다.
4. Layer 1만 제공하는 기능은 경쟁력이 없다.
```

## 3. 행동 과학 원리

### 3.1 자기결정 이론 (Self-Determination Theory, Deci & Ryan)

인간의 내재적 동기는 세 가지 기본 욕구에서 나온다:

| 욕구                     | 정의                 | 이룸 적용                  |
| ------------------------ | -------------------- | -------------------------- |
| **자율성** (Autonomy)    | 스스로 선택하는 느낌 | 추천은 선택지, 지시가 아님 |
| **유능감** (Competence)  | 성장하는 느낌        | 판단력 성장 가시화         |
| **관계성** (Relatedness) | 연결된 느낌          | 공감적 AI 코치 톤          |

**설계 규칙:**

- 자율성: "이걸 해보세요" (X) → "이런 이유로 이 옵션이 있어요" (O)
- 유능감: "AI가 추천합니다" (X) → "지난번에 직접 고른 컬러가 정확했어요" (O)
- 관계성: 기계적 톤 (X) → 따뜻하지만 과하지 않은 톤 (O)

### 3.2 Fogg 행동 모델 (B=MAP)

```
행동(Behavior) = 동기(Motivation) × 능력(Ability) × 프롬프트(Prompt)
```

- **동기**: 이미 존재 (앱을 깔았으므로)
- **능력**: 이룸의 핵심 병목 — 사용자가 "왜 이게 나한테 맞는지" 모름
- **프롬프트**: 적절한 타이밍에 적절한 인사이트

**이룸의 역할 = 능력(Ability) 향상**

- 교육이 아닌 경험으로 능력 향상
- "봄 웜톤 이론을 가르친다" (X) → "코랄 립 추천 시 '웜톤이라서'를 보여준다" (O)

### 3.3 연결 인식 (Connection Awareness)

이룸 고유의 자기 이해 원자 단위:

```
연결 = "A라서 B"

예시:
- "봄 웜톤이라서 코랄 립이 어울려요"
- "건성 피부라서 히알루론산 세럼이 좋아요"
- "상체 발달형이라서 V넥이 균형을 잡아줘요"
```

#### 연결 상태 모델 (ConnectionStatus)

```
exposed      → 첫 노출 ("봄 웜톤이라서 코랄이 어울려요"를 처음 봄)
recognized   → 인식 (3회+ 노출, "아, 웜톤이라 그렇구나" 이해)
internalized → 내재화 (5회+ 확인, 스스로 "웜톤이니까..." 판단)
independent  → 자립 (7회+, AI 없이 직접 판단)
```

#### 내재화 곡선

| 노출 횟수 | 상태   | 사용자 반응                            |
| --------- | ------ | -------------------------------------- |
| 1회       | 정보   | "그렇구나" (수동적 수용)               |
| 3회+      | 인식   | "아, 이래서!" (능동적 이해)            |
| 5회+ 확인 | 내재화 | "웜톤이니까 이게 맞겠다" (자발적 적용) |
| 7회+      | 자립   | AI 설명 없이 직접 판단                 |

#### 데이터 모델

```typescript
interface ConnectionAwareness {
  connectionId: string; // "pc-warm-coral-lip"
  sourceModule: AnalysisModule; // 'personal-color'
  targetDomain: string; // 'makeup'
  connectionRule: string; // "A라서 B" 형태
  exposureCount: number; // 노출 횟수
  confirmedCount: number; // 사용자가 확인/수용한 횟수
  status: ConnectionStatus; // exposed | recognized | internalized | independent
  lastExposedAt: Date;
}
```

## 4. Aha Moment

### 4.1 정의

```
Aha Moment = 2번째 분석 완료 + 첫 교차 인사이트 수신

예: 퍼스널컬러 분석 후 피부 분석 → "봄 웜톤 + 건성이시네요,
    이 조합에는 웜톤 계열 보습 세럼이 특히 잘 맞아요"
```

### 4.2 왜 2번째인가

- 1번째 분석: Layer 1 (즉각 해결) — 단일 결과
- 2번째 분석: Layer 2 진입 — 교차 인사이트 가능
- 교차 인사이트 = 이룸만의 가치 = "통합된 자기 이해"의 시작

### 4.3 핵심 KPI

```
Primary KPI: 2번째 분석 완료율 (2nd Analysis Completion Rate)

이유:
- 1번째 완료 = 호기심 (모든 앱이 가능)
- 2번째 완료 = 가치 인식 (이룸만의 교차 인사이트 경험)
- 3번째부터 = 습관화 (리텐션 예측)
```

## 5. 이해 엔진 아키텍처

### 5.1 구조

```
+--------------------------------------------------+
|           이해 엔진 (Understanding Engine)          |
+--------------------------------------------------+
|                                                    |
|   +--------------+    +--------------+            |
|   |  Rule Engine  |    |   Gemini AI  |            |
|   |  "무엇을"     |    |   "왜를"     |            |
|   |              |    |              |            |
|   |  교차 인사이트 |    |  자연어 설명  |            |
|   |  연결 규칙     |    |  개인화 맥락  |            |
|   +------+-------+    +------+-------+            |
|          |                    |                     |
|          +--------+-----------+                     |
|                   v                                 |
|          +----------------+                         |
|          |  Safety Layer  |                         |
|          |  의학적 검증    |                         |
|          |  과장 방지     |                         |
|          +--------+-------+                         |
|                   v                                 |
|          +----------------+                         |
|          |  Progression   |                         |
|          |  Tracker       |                         |
|          |  내재화 추적    |                         |
|          +----------------+                         |
|                                                    |
+--------------------------------------------------+
```

### 5.2 각 컴포넌트 역할

| 컴포넌트                | 역할                               | 기술                 |
| ----------------------- | ---------------------------------- | -------------------- |
| **Rule Engine**         | 교차 인사이트 규칙 매칭            | lib/insights/ (기존) |
| **Gemini AI**           | "왜" 자연어 설명 생성              | Gemini 3 Flash API   |
| **Safety Layer**        | 의학적/과학적 검증, 과장 방지      | 규칙 기반 필터       |
| **Progression Tracker** | ConnectionStatus 추적, 내재화 곡선 | Supabase             |

### 5.3 Rule Engine vs Gemini 역할 분담

```
Rule Engine이 하는 것:
- "봄 웜톤 + 건성 → 보습 웜톤 세럼 추천" (규칙 매칭)
- "피부 점수 변화 → 개선/악화 판정" (통계 비교)
- "교차 인사이트 후보 생성" (모듈 간 연결)

Gemini가 하는 것:
- "왜 웜톤에 코랄이 어울리는지" 자연어 설명
- 사용자 맥락에 맞는 개인화 표현
- 복잡한 다중 모듈 인사이트 종합

둘 다 하지 않는 것:
- 의학적 진단 (면책 필요)
- 확정적 처방 ("반드시 ~하세요")
- 타인과의 비교
```

## 6. 감정 설계 (Emotional Design)

### 6.1 감정 원칙

| 원칙              | 설명                             | 예시                                 |
| ----------------- | -------------------------------- | ------------------------------------ |
| **호기심 → 이해** | 궁금함에서 시작, 납득으로 마무리 | "왜 이게 나한테 맞을까?" → "이래서!" |
| **발견의 기쁨**   | 모르던 것을 알게 되는 긍정 경험  | "내가 이런 타입이었구나!"            |
| **성장 실감**     | 과거와 현재 비교로 변화 인식     | "3개월 전보다 수분 점수가 올랐어요"  |
| **자신감 강화**   | 자기 판단이 맞았을 때 확인       | "직접 고른 컬러가 정확했어요!"       |

### 6.2 금지 감정

| 감정       | 왜 금지하는가            | 대안                                 |
| ---------- | ------------------------ | ------------------------------------ |
| **불안**   | "이거 안 하면 큰일 나요" | "이걸 해보면 더 좋아질 수 있어요"    |
| **열등감** | "평균보다 낮아요"        | "현재 상태에서 이런 개선이 가능해요" |
| **강박**   | "매일 해야 해요"         | "여유 있을 때 해보세요"              |
| **비교**   | "다른 사용자는..."       | 절대 타인 데이터 노출 금지           |

### 6.3 신체 이미지 보호 (Body Image Protection)

```
절대 원칙:
1. 결핍 프레이밍 금지 — "부족하다", "안 좋다" 표현 사용 안 함
2. 타인 비교 금지 — "평균", "이상적", "표준" 체형 언급 안 함
3. 부정적 신체 언어 금지 — "문제 부위", "약점" 표현 사용 안 함
4. 항상 긍정적 행동으로 마무리 — 분석 결과 후 실행 가능한 긍정 액션 제시

예시:
X "엉덩이가 좁은 편이에요"
O "직선적인 실루엣이 특징이에요. A라인 스커트가 잘 어울려요"

X "피부 상태가 안 좋아요"
O "수분 보충에 집중하면 더 좋아질 수 있어요"
```

## 7. 경쟁 해자 (Competitive Moat)

### 7.1 해자 공식

```
이룸의 해자 = 분석 모듈 수 × 교차 인사이트 품질

현재: 9 모듈 × 6종 인사이트 = 54+ 가능 조합
경쟁: 단일 분석 앱은 교차 인사이트 생성 불가
```

### 7.2 해자 강화 전략

| 전략              | 설명                    | 방어력           |
| ----------------- | ----------------------- | ---------------- |
| **모듈 수 확장**  | 새 분석 모듈 추가       | 조합 폭발적 증가 |
| **인사이트 깊이** | 교차 인사이트 품질 향상 | 단순 복제 불가   |
| **내재화 데이터** | ConnectionStatus 축적   | 시간 기반 장벽   |
| **개인화 맥락**   | 사용자별 히스토리       | 전환 비용        |

### 7.3 경쟁 앱이 모방할 수 없는 것

```
1. 9모듈 통합 분석 인프라 (구축 비용 높음)
2. 교차 인사이트 규칙 엔진 (도메인 전문성 필요)
3. 내재화 곡선 데이터 (시간 축적 필요)
4. "판단력 성장" 설계 철학 (기술이 아닌 철학의 차이)
```

## 8. 비즈니스 정렬 (Business Alignment)

### 8.1 수익 모델 원칙

```
수익은 사용자 자립을 방해하지 않아야 한다.
```

| 수익 모델         | 정렬 여부 | 이유                              |
| ----------------- | --------- | --------------------------------- |
| **어필리에이트**  | 정렬      | 추천 = 이해 기반 선택 지원        |
| **프리미엄 분석** | 정렬      | 더 깊은 분석 = 더 깊은 이해       |
| **구독**          | 조건부    | 가치 제공 시에만 (잠금 장치 아님) |
| **데이터 판매**   | 영구 금지 | 사용자 신뢰 파괴                  |
| **광고**          | 금지      | 주의 경제 ≠ 이해 경제             |

### 8.2 어필리에이트 정렬 조건

```
추천 제품에 항상 "왜" 포함:
- "봄 웜톤이라서 이 코랄 립이 어울려요" (O)
- "인기 상품이에요" (X — 이해와 무관)
- "할인 중이에요" (X — 가격은 부가 정보)

추천 순서: 적합도(CCS) > 사용자 선호 > 가격/인기
```

### 8.3 성장-자립 역설 해결

```
역설: 사용자가 자립하면 앱을 안 쓴다 → 성장이 멈춘다?

해결:
1. 자립한 사용자 = 최고의 추천자 (입소문)
2. 새 모듈 = 새로운 Layer 1 진입 (웜톤은 알지만 체형은 모름)
3. 계절/트렌드 변화 = 주기적 재방문 (봄 팔레트 → 여름 팔레트)
4. 깊이 확장 = 기존 모듈의 고급 분석 추가
```

## 9. 윤리 원칙 (Ethics)

### 9.1 핵심 윤리

| 원칙       | 규칙                            | 위반 시                  |
| ---------- | ------------------------------- | ------------------------ |
| **투명성** | AI 분석임을 항상 고지           | 의학 진단으로 오해 유발  |
| **비의료** | "진단"이 아닌 "분석"으로 표현   | 법적 리스크              |
| **동의**   | 이미지 사용 범위 명시적 동의    | 신뢰 파괴                |
| **삭제권** | 언제든 데이터 완전 삭제 가능    | GDPR/개인정보보호법 위반 |
| **비차별** | 피부색/성별/체형 편향 없는 분석 | 차별 조장                |

### 9.2 AI 한계 고지

```
모든 AI 분석 결과에 포함:
"이 결과는 AI 기반 분석으로, 전문가 진단을 대체하지 않습니다."

건강 관련 인사이트에 추가:
"정확한 진단은 전문의와 상담해주세요."
```

### 9.3 데이터 윤리

```
수집: 분석에 필요한 최소한의 데이터만
저장: 암호화 + RLS + 사용자별 격리
활용: 해당 사용자의 인사이트 생성에만
공유: 제3자 공유 절대 금지
삭제: 사용자 요청 시 30일 내 완전 삭제
```

## 11. 측정 프레임워크

### 11.1 핵심 지표 (North Star)

```
North Star Metric: 월간 자립적 판단 횟수
= AI 설명 없이 사용자가 직접 선택한 횟수

예: 추천 없이 직접 제품 선택, 직접 운동 계획 수정
```

### 11.2 Layer별 지표

| Layer     | 지표                    | 목표    | 측정 방법             |
| --------- | ----------------------- | ------- | --------------------- |
| **L1**    | 첫 분석 완료율          | 80%+    | 온보딩 퍼널           |
| **L1→L2** | 2번째 분석 완료율 (Aha) | 50%+    | 퍼널 분석             |
| **L2**    | 교차 인사이트 확인율    | 60%+    | 클릭/체류             |
| **L2→L3** | 연결 내재화율           | 30%+    | ConnectionStatus 통계 |
| **L3**    | 자립적 판단 빈도        | 월 5회+ | 비추천 선택 로그      |

### 11.3 반지표 (Anti-Metrics)

```
경계해야 할 지표:
- DAU/MAU 단독 → 의존도 높아도 올라감
- 세션 시간 → 길다고 좋은 것 아님
- 추천 수락률 단독 → 맹목적 수락 구분 불가
- 알림 클릭률 → 알림 남발로 올릴 수 있음
```

## 12. 온보딩 설계

### 12.1 Aha Moment 최적화

```
온보딩 목표: 가능한 빨리 2번째 분석 완료시키기

플로우:
1. 첫 분석 (퍼스널컬러 — 가장 접근성 높음)
2. 결과 화면에서 "피부 분석도 해보면 더 정확한 추천이 가능해요" 유도
3. 2번째 분석 완료 → 첫 교차 인사이트 → Aha!
4. "두 분석을 연결하니 이런 인사이트가 나왔어요" 강조
```

### 12.2 단계별 가치 전달

| 단계         | 사용자 상태 | 이룸의 메시지                            |
| ------------ | ----------- | ---------------------------------------- |
| 설치         | 호기심      | "나에 대해 알아보세요"                   |
| 첫 분석      | 신기함      | "당신은 [결과]이에요"                    |
| 두 번째 분석 | 연결 발견   | "[A]라서 [B]가 어울려요"                 |
| 세 번째+     | 패턴 인식   | "이제 이런 부분도 직접 판단할 수 있어요" |

## 15. 설계 판단 체크리스트

모든 기능/변경에 이 체크리스트를 적용한다:

```
[ ] 이 기능은 어떤 Layer에 기여하는가? (L1/L2/L3)
[ ] Layer 2 이상에 기여하지 않으면, 왜 필요한가?
[ ] 사용자의 자립을 돕는가, 의존을 만드는가?
[ ] "왜"를 설명하는가, 결과만 제시하는가?
[ ] 선택지를 주는가, 지시를 내리는가?
[ ] 긍정적 감정을 유발하는가? (호기심, 발견, 성장, 자신감)
[ ] 금지 감정을 유발하지 않는가? (불안, 열등감, 강박, 비교)
[ ] 신체 이미지 보호 원칙을 준수하는가?
[ ] 수익 모델이 자립을 방해하지 않는가?
[ ] 윤리 원칙(투명성, 비의료, 동의, 삭제권, 비차별)을 준수하는가?
```
