# 캡슐 원리 (Capsule Principles)

> 이 문서는 캡슐 에코시스템의 기반이 되는 5가지 원리를 정의한다.
> 9개 분석 모듈(PC, S, C, W, N, H, M, OH, Fashion)에 걸쳐 적용되는 범용 큐레이션 프레임워크이다.
>
> **소스 리서치**: Miller's Law (1956), 선택의 역설 (Schwartz, 2004), 만족화 전략 (Simon, 1956)
> **관련 원리**: [cross-domain-synergy.md](./cross-domain-synergy.md), [personalization-engine.md](./personalization-engine.md), [fashion-matching.md](./fashion-matching.md)

## 0. 궁극의 형태 (P1)

### 이상적 최종 상태

```
"완벽한 통합 캡슐 시스템"

- 9개 도메인 통합 캡슐: 모든 분석 결과가 하나의 캡슐 에코시스템으로 수렴
- 최적 N 자동 결정: 사용자 숙련도/모듈 복잡도에 따라 추천 개수 동적 조절
- 100% 상호 호환: 캡슐 내 모든 아이템이 서로 시너지
- 시간축 적응: 계절, 피부 주기, 운동 적응에 따라 자동 로테이션
- 의사결정 피로 제로: 사용자가 고민 없이 실행 가능한 액션 제공
- 설명 가능: "왜 이 N개인지" 각 선택에 과학적 근거 제공
```

### 물리적 한계

| 한계               | 설명                                                      |
| ------------------ | --------------------------------------------------------- |
| **개인 차이**      | 동일 분석 결과라도 개인 선호/체질에 따라 최적 캡슐이 상이 |
| **데이터 희소성**  | 신규 사용자는 히스토리 부재로 로테이션/적응 제한          |
| **도메인 간 충돌** | 피부에 좋은 성분이 영양학적으로 부적합할 수 있음          |
| **시간 지연**      | 영양 변화 → 피부 개선 2~4주, 운동 → 체형 변화 4~8주       |
| **트렌드 변동**    | 패션/뷰티 트렌드가 과학적 최적화와 충돌 가능              |

### 100점 기준

| 지표                | 100점 기준                         |
| ------------------- | ---------------------------------- |
| **큐레이션 정확도** | 추천 아이템 만족도 4.5/5.0 이상    |

| **의사결정 시간**   | 캡슐 제시 후 실행까지 평균 < 30초  |
| **도메인 커버리지** | 9개 도메인 모두 캡슐화             |

## 1. 핵심 개념

### 1.1 캡슐의 정의

"무한한 선택지에서 개인에게 최적화된 소수의 핵심 아이템을 선별하고,
이들의 상호 호환성을 보장하며, 주기적으로 갱신하는 시스템"

```
캡슐 시스템 = 큐레이션(C1) × 호환성(C2) × 개인화(C3) × 로테이션(C4) × 미니멀리즘(C5)

각 원리는 독립적으로 적용 가능하지만,
5가지가 함께 작동할 때 최대 가치를 제공한다.
```

### 1.2 캡슐이 해결하는 문제

| 문제            | 근거                                                               | 캡슐의 해결                   |
| --------------- | ------------------------------------------------------------------ | ----------------------------- |
| **선택 과부하** | 선택의 역설 (Schwartz, 2004) — 선택지가 많을수록 만족도 하락       | 핵심 N개로 축소               |
| **결정 피로**   | 의사결정 피로 (Baumeister et al., 1998) — 반복 결정이 질 저하 초래 | 사전 큐레이션으로 결정 최소화 |
| **비호환 조합** | 개별 최적 ≠ 조합 최적 — 좋은 아이템도 조합이 나쁘면 효과 감소      | 호환성 점수로 조합 보장       |
| **고착/적응**   | 피부 내성, 근육 적응 — 동일 루틴 반복 시 효과 감소                 | 로테이션으로 주기적 갱신      |
| **과잉 축적**   | 미니멀리즘 결여 — 불필요 아이템 축적으로 관리 비용 증가            | "적을수록 좋다" 원칙          |

## 2. 5가지 원리

### C1: 큐레이션 (Curation)

> "무한한 선택지에서 핵심 N개만 남긴다"

#### 2.1.1 이론적 근거

**밀러의 법칙 (Miller, 1956)**: 인간의 작업 기억 용량은 7±2개 정보 단위(chunk)이다.
한 번에 처리할 수 있는 선택지가 이를 초과하면 인지 부하가 급격히 증가한다.

**선택의 역설 (Schwartz, 2004; Iyengar & Lepper, 2000)**:
Iyengar와 Lepper의 잼 실험에서 24종 진열 시 구매율 3%, 6종 진열 시 구매율 30%로
선택지 축소가 행동 전환율을 10배 향상시켰다.

```
핵심 공식:

  Optimal_N = f(module_complexity, user_expertise)

  여기서:
  - module_complexity: 해당 도메인의 아이템 다양성 (1-10)
  - user_expertise: 사용자 숙련도 레벨 (L1-L4)

  실용 범위: 3 ≤ Optimal_N ≤ 10
```

#### 2.1.2 숙련도별 N값 결정

| 숙련도        | 정의                  | N값 범위  | 근거                                   |
| ------------- | --------------------- | --------- | -------------------------------------- |
| **L1 입문자** | 해당 도메인 경험 없음 | 3~5       | 밀러 하한(5)보다 낮게 — 최소 인지 부하 |
| **L2 중급**   | 기본 지식 보유        | 5~7       | 밀러 중앙값 — 적정 선택 범위           |
| **L3 고급**   | 도메인 전문 지식      | 7~10      | 밀러 상한(9) 근처 — 세분화 선택 가능   |
| **L4 전문가** | 자체 판단 가능        | 제한 없음 | 큐레이션 비활성화, 전체 목록 제공      |

#### 2.1.3 모듈별 기본 N값

| 모듈            | 복잡도 | L1 (입문) | L2 (중급) | L3 (고급) | 설명               |
| --------------- | ------ | --------- | --------- | --------- | ------------------ |
| PC-1 퍼스널컬러 | 5      | 4색       | 6색       | 8색       | 핵심 컬러 팔레트   |
| S-1 피부        | 7      | 3제품     | 5제품     | 7제품     | 스킨케어 루틴 스텝 |
| C-1 체형        | 4      | 3종       | 5종       | 7종       | 추천 실루엣 유형   |
| W-1 운동        | 6      | 3종목     | 5종목     | 7종목     | 주간 루틴 종목     |
| N-1 영양        | 8      | 3항목     | 5항목     | 8항목     | 핵심 영양소/보충제 |
| H-1 헤어        | 4      | 3제품     | 4제품     | 6제품     | 헤어 관리 루틴     |
| M-1 메이크업    | 6      | 4제품     | 6제품     | 8제품     | 메이크업 파우치    |
| OH-1 구강       | 3      | 3제품     | 4제품     | 5제품     | 구강 관리 루틴     |
| J-1 패션        | 9      | 5아이템   | 7아이템   | 10아이템  | 캡슐 옷장 핵심     |

### C2: 호환성 (Compatibility)

> "선택된 아이템 간 상호 매칭을 보장한다"

#### 2.2.1 이론적 근거

개별 최적(individually optimal)이 조합 최적(jointly optimal)을 보장하지 않는다.
캡슐의 가치는 아이템 간 시너지에서 나온다.

```
호환성 점수 (Capsule Compatibility Score):

  CCS = Σ(pairwise_compatibility(i, j)) / C(n, 2)

  여기서:
  - pairwise_compatibility(i, j): 아이템 i와 j의 쌍별 호환성 (0-100)
  - C(n, 2) = n(n-1)/2: 가능한 쌍의 수 (조합 수)
  - n: 캡슐 내 아이템 수

  예: 캡슐에 5개 아이템 → C(5,2) = 10쌍 평가
  합격 기준: CCS ≥ 70
```

#### 2.2.2 모듈별 호환성 차원

| 모듈                | 호환성 차원                         | 측정 방법                     | 합격 기준            |
| ------------------- | ----------------------------------- | ----------------------------- | -------------------- |
| **PC-1 퍼스널컬러** | 색상 조화 (Delta-E in Lab space)    | ΔE₀₀ < 15: 조화, > 30: 부조화 | ΔE₀₀ ≤ 20            |
| **S-1 피부**        | 성분 상호작용 (synergy vs conflict) | 시너지 매트릭스 조회          | 충돌 성분 0개        |
| **C-1 체형**        | 실루엣 균형 (상/하 비율)            | 비율 점수 0-100               | ≥ 70                 |
| **W-1 운동**        | 근육군 밸런스 (주동근/길항근)       | 근육군 커버리지               | 6대 근육군 모두 포함 |
| **N-1 영양**        | 영양소 보완성 (결핍 보충)           | 영양소 커버리지 맵            | 주요 결핍 100% 커버  |
| **H-1 헤어**        | 두피-모발 적합성                    | 성분-두피타입 매칭            | 충돌 성분 0개        |
| **M-1 메이크업**    | 컬러 하모니 + 제형 레이어링         | 퍼스널컬러 내 색상            | 전체 캡슐 동일 톤    |
| **OH-1 구강**       | 성분 순서 (치약 → 구강청결제)       | 사용 순서 호환성              | 순서 충돌 0건        |
| **J-1 패션**        | 캡슐 옷장 조합률                    | 상호 매칭 가능 비율           | ≥ 80% 조합 가능      |

#### 2.2.3 성분 호환성 예시 (S-1 스킨케어)

```
시너지 조합 (함께 사용 시 효과 ↑):
  나이아신아마이드 + 히알루론산 = 보습+미백 시너지
  비타민C + 비타민E = 항산화 시너지 (1+1 > 2)
  레티놀 + 세라마이드 = 재생+보습 보완

충돌 조합 (함께 사용 시 효과 ↓ 또는 자극):
  레티놀 + AHA/BHA = 과도한 각질 제거 → 자극 위험
  비타민C + 나이아신아마이드 = pH 충돌 (저농도 시 허용)
  벤조일퍼옥사이드 + 레티놀 = 상호 비활성화

호환성 판정:
  synergy → +20점 보너스
  neutral → 0점
  conflict → 아이템 교체 권장 (캡슐에서 제외)
```

#### 2.2.4 크로스 도메인 호환성

캡슐은 단일 모듈 내부뿐 아니라 모듈 간 호환성도 고려한다.

```
크로스 도메인 호환성 예시:

  S-1(피부) × N-1(영양):
    건성 피부 → 오메가-3, 비타민E 보충제 캡슐 포함

  W-1(운동) × N-1(영양):
    고강도 운동 루틴 → 단백질 보충제 캡슐 포함

  PC-1(퍼스널컬러) × M-1(메이크업):
    봄 웜톤 → 코랄 립 + 피치 블러셔 호환

  PC-1(퍼스널컬러) × J-1(패션):
    퍼스널컬러 팔레트 내 의류 색상만 캡슐 포함
```

### C3: 개인화 (Personalization)

> "분석 결과에 기반하여 개인에게 최적화한다"

#### 2.3.1 이론적 근거

범용 추천은 개인에게 차선(suboptimal)이다.
캡슐 시스템의 핵심 가치는 "당신만을 위한" 선별이다.

```
개인화 점수 = f(프로필 깊이, 행동 데이터, 분석 결과)

프로필 깊이: 완료한 분석 모듈 수 (0-9)
행동 데이터: 구매/사용/피드백 이력
분석 결과: 각 모듈의 정량적 분석 값
```

#### 2.3.2 개인화 레벨

| 레벨   | 이름      | 데이터 소스         | 캡슐 정밀도 | 예시                                     |
| ------ | --------- | ------------------- | ----------- | ---------------------------------------- |
| **L1** | 기본 분류 | 분석 결과 카테고리  | 60%         | "건성 피부" → 보습 제품군                |
| **L2** | 세부 점수 | 분석 세부 수치      | 75%         | "건성(수분도 35)" → 고보습 세럼 특정     |
| **L3** | 히스토리  | 과거 캡슐 사용 이력 | 85%         | "레티놀 사용 3개월" → 농도 업그레이드    |
| **L4** | 행동 패턴 | 구매/리뷰/재구매    | 95%         | "좋아요 누른 제형 패턴" → 질감 선호 반영 |

#### 2.3.3 BeautyProfile 통합

9개 분석 모듈의 결과가 하나의 BeautyProfile로 통합되며,
캡슐 개인화의 데이터 소스 역할을 한다.

> **정식 TypeScript 정의**: [ADR-069](../adr/ADR-069-capsule-ecosystem-architecture.md) §BeautyProfile 참조

```typescript
type ModuleCode = 'PC' | 'S' | 'C' | 'W' | 'N' | 'H' | 'M' | 'OH' | 'Fashion';

interface BeautyProfile {
  userId: string;
  updatedAt: string;

  // 9모듈 분석 결과 (각 모듈 완료 시 자동 갱신, 미완료 시 undefined)
  personalColor?: { season: string; subType: string; palette: string[] };
  skin?: { type: string; concerns: string[]; scores: Record<string, number> };
  body?: { shape: string; measurements: Record<string, number> };
  workout?: { fitnessLevel: string; goals: string[]; history: string[] };
  nutrition?: { deficiencies: string[]; dietType: string; allergies: string[] };
  hair?: { type: string; scalp: string; concerns: string[] };
  makeup?: { preferences: Record<string, string>; skillLevel: string };
  oral?: { conditions: string[]; goals: string[] };
  fashion?: { style: string; sizeProfile: Record<string, string>; wardrobe: string[] };

  // 메타데이터
  completedModules: ModuleCode[];
  personalizationLevel: 1 | 2 | 3 | 4; // L1~L4 (위 표 참조)
  lastFullUpdate: string;
}

// 개인화 정밀도 = completedModules.length / 9 × 가중 평균(레벨별 정밀도)
```

### C4: 로테이션 (Rotation)

> "캡슐은 정적이지 않다. 주기적으로 갱신한다"

#### 2.4.1 이론적 근거

**피부 내성 (Tachyphylaxis)**: 동일 활성 성분에 장기 노출 시 반응 감소.
레티놀, AHA 등 일부 성분은 4~8주 후 피부가 적응하여 효과 정체.

**근육 적응 (Muscle Adaptation)**: 동일 운동 프로그램 4~6주 후 성장 정체.
SAID 원칙(Specific Adaptation to Imposed Demands)에 의해 변화 필요.

**계절 변화**: 기온/습도 변화에 따라 피부 상태, 의류 필요, 영양 요구가 달라짐.

**심리적 신선함 (Novelty Effect)**: 동일 루틴의 지루함은 이탈(churn)의 주요 원인.

#### 2.4.2 모듈별 교체 주기

| 모듈                | 기본 주기   | 트리거 조건               | 갱신 범위              |
| ------------------- | ----------- | ------------------------- | ---------------------- |
| **PC-1 퍼스널컬러** | 계절(3개월) | 시즌 변경, 피부톤 변화    | 팔레트 30% 교체        |
| **S-1 피부**        | 4~8주       | 피부 상태 변화, 성분 내성 | 액티브 성분 1~2개 교체 |
| **C-1 체형**        | 3~6개월     | 체중/체형 변화            | 추천 실루엣 갱신       |
| **W-1 운동**        | 4~6주       | 성장 정체, 부상           | 종목 30~50% 교체       |
| **N-1 영양**        | 4~12주      | 혈액 검사, 목표 변경      | 보충제 재평가          |
| **H-1 헤어**        | 8~12주      | 계절, 두피 상태           | 제품 1~2개 교체        |
| **M-1 메이크업**    | 계절(3개월) | 트렌드, 퍼스널컬러 변동   | 포인트 컬러 교체       |
| **OH-1 구강**       | 12주        | 검진 결과                 | 치약/구강청결제 교체   |
| **J-1 패션**        | 계절(3개월) | 시즌 체인지               | 캡슐 옷장 30% 교체     |

#### 2.4.3 로테이션 알고리즘

```
로테이션 판정:

  rotation_needed = (
    time_since_last_update > module_cycle
    OR trigger_condition_met
    OR user_requested
  )

  교체 범위 결정:
  - 경미한 변화: 캡슐의 20~30% 교체 (유지 기반)
  - 중간 변화: 캡슐의 30~50% 교체 (진화 기반)
  - 대규모 변화: 캡슐의 50%+ 교체 (리셋 기반)

  교체 대상 선정:
  1. 호환성 점수 최하위 아이템 우선 교체
  2. 사용자 피드백 부정 아이템 교체
  3. 트리거 조건 직접 관련 아이템 교체
```

### C5: 미니멀리즘 (Minimalism)

> "적을수록 좋다. 충분히 좋은 것이 최선이다"

#### 2.5.1 이론적 근거

**의사결정 피로 (Decision Fatigue, Baumeister et al., 1998)**:
의사결정은 유한한 인지 자원을 소모한다. 하루 평균 35,000건의 결정 중
뷰티/건강 결정을 최소화하면 다른 영역에 인지 자원을 보존할 수 있다.

**기본값 효과 (Default Effect)**:
기본값(default)이 제시되면 70~90%의 사용자가 변경하지 않는다 (Johnson & Goldstein, 2003).
캡슐이 "좋은 기본값"을 제공하면 사용자는 최소 노력으로 최적에 근접할 수 있다.

**만족화 전략 (Satisficing, Herbert Simon, 1956)**:
최적화(maximizing)보다 만족화(satisficing)가 더 높은 만족도를 달성한다.
"충분히 좋은(good enough)" 선택이 "완벽한" 선택을 찾는 것보다 실용적이다.

#### 2.5.2 미니멀리즘 원칙 적용

```
1. 추가보다 제거를 먼저 고려한다
   - 새 아이템 추가 시 기존 아이템 제거 검토
   - "이것 없이도 목표 달성 가능한가?"

2. 기본 캡슐은 최소 크기에서 시작한다
   - L1 숙련도 기준 N값으로 시작
   - 사용자가 요청할 때만 확장

3. 하나의 아이템이 여러 역할을 하면 우선 선택한다
   - 멀티 기능 제품 우대
   - 단일 기능 제품보다 다기능 제품이 캡슐 효율↑

4. "충분히 좋은" 임계값을 설정한다
   - 매칭 점수 75 이상이면 "충분히 좋음"
   - 95점 제품과 80점 제품의 실질 차이가 미미하면 80점 우선 (비용 효율)
```

#### 2.5.3 캡슐 크기 상한

| 모듈 | 절대 상한 | 근거                      |
| ---- | --------- | ------------------------- |
| PC-1 | 10색      | 계절 팔레트 최대 확장     |

| C-1  | 8종       | 의류 카테고리 수 제한     |
