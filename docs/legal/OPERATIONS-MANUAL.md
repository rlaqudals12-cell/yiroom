## 1. 사고 분류

### 1.1 심각도 등급

| 등급              | 정의                      | 예시                                    | 대응 시한       |
| ----------------- | ------------------------- | --------------------------------------- | --------------- |
| **P0 (Critical)** | 사용자 건강에 직접적 위험 | 알레르겐 미탐지(FNR), 금기 성분 미차단  | 즉시 (1시간 내) |
| **P1 (High)**     | 민감정보 노출 / 법적 위반 | 암호화 실패, 동의 없이 수집, 로그에 PII | 4시간 내        |
| **P2 (Medium)**   | 기능 오작동 / 오정보      | EWG 등급 오표시, 잘못된 Safety 점수     | 24시간 내       |
| **P3 (Low)**      | UX 문제 / 경미한 버그     | 면책 문구 미표시, 로딩 지연             | 72시간 내       |

### 1.2 사고 유형

```
Safety 사고:
  - 알레르겐 미탐지 (Level 1 BLOCK 실패)
  - 금기사항 미차단 (임산부 금기 성분 통과)
  - 성분 상호작용 미탐지
  - Safety 점수 오계산

개인정보 사고:
  - 민감정보 암호화 실패
  - 동의 없이 Safety Profile 수집
  - 로그에 복호화된 민감정보 기록
  - RLS 우회로 타인 데이터 접근
  - 동의 철회 후 데이터 미삭제

법적 사고:
  - 의료 조언으로 해석 가능한 표현 사용
  - 금지 표현 사용 ("치료", "효과", "의사 추천")
  - 면책 문구 누락
  - 어필리에이트 공시 누락

시스템 사고:
  - Safety API 전면 장애
  - 암호화 키 유출
  - DB 장애로 Safety 검사 불가
```

## 2. 사고 대응 플로우

### 2.1 P0 (Critical) — Safety 오검증

```
[발견] → 즉시 시작
  │
  ├── 0~15분: 긴급 조치
  │   ├── 해당 알레르겐/금기 규칙 긴급 패치
  │   ├── Feature Flag로 해당 Safety 검사 일시 비활성화
  │   └── 로그: "[INCIDENT-P0] Safety misdetection: {type}"
  │
  ├── 15분~1시간: 영향 범위 파악
  │   ├── 영향 받은 사용자 식별 (DB 쿼리)
  │   ├── 해당 SafetyReport 캐시 전체 무효화
  │   └── 영향 사용자에게 인앱 알림 발송
  │
  ├── 1~24시간: 근본 원인 분석
  │   ├── 원인: 규칙 누락? DB 오류? 코드 버그?
  │   ├── 수정 코드 작성 + 테스트
  │   └── 핫픽스 배포
  │
  └── 24~72시간: 재발 방지
      ├── docs/incidents/ 사고 보고서 작성
      ├── 테스트 케이스 추가 (해당 시나리오)
      └── FNR 모니터링 강화
```

### 2.2 P1 (High) — 개인정보 사고

```
[발견] → 4시간 내
  │
  ├── 0~30분: 긴급 조치
  │   ├── 노출 경로 차단 (API 비활성화, 로그 삭제)
  │   ├── 영향 범위 파악 (노출된 사용자 수, 데이터 종류)
  │   └── 로그: "[INCIDENT-P1] Privacy breach: {type}"
  │
  ├── 30분~4시간: 법적 대응 판단
  │   ├── 개인정보보호법 제34조 통지 의무 검토
  │   │   └── 1,000명+ 또는 민감정보 → 개인정보보호위원회 72시간 내 신고
  │   ├── 영향 사용자에게 개별 통지 준비
  │   └── 수정 조치 실행
  │
  ├── 4~24시간: 통지 실행
  │   ├── 개인정보보호위원회 신고 (해당 시)
  │   ├── 영향 사용자 통지 (이메일 + 인앱)
  │   └── 통지 내용: 사고 개요, 노출 범위, 조치 사항, 연락처
  │
  └── 24~72시간: 재발 방지
      ├── docs/incidents/ 사고 보고서 작성
      ├── 암호화/접근제어 강화 조치
      └── 보안 감사 실행
```

### 2.3 P2/P3 — 일반 사고

```
[발견]
  ├── 이슈 등록 (GitHub Issues)
  ├── 영향 범위 파악
  ├── 수정 + 테스트 + 배포
  └── 재발 방지 문서화
```

## 3. 정기 모니터링

### 3.1 Safety 모니터링 일정

| 주기 | 항목                        | 방법                | 기준      |
| ---- | --------------------------- | ------------------- | --------- |
| 매일 | Safety API 헬스체크         | /api/health/safety  | 200 OK    |
| 매주 | 알레르겐 DB 교차 검증       | 100건 샘플 테스트   | FNR 0%    |

| 주기 | 항목                     | 방법                        |
| ---- | ------------------------ | --------------------------- |

| 매주 | RLS 정책 유효성          | Supabase RLS 테스트         |
| 매월 | 동의 철회 처리 완료 확인 | safety_profiles 삭제 기록   |
| 분기 | 12개월 미접속 익명화     | 대상 사용자 식별 + 실행     |

| 주기 | 항목                      | 방법                               |

## 4. 금지 표현 모니터링

### 4.1 자동 스캔 대상 키워드

```
금지 표현 (코드 + UI 텍스트):
  - "치료합니다", "치료해요", "치료"
  - "효과가 있습니다", "효과가 있어요"
  - "의사가 추천", "피부과 전용", "의사 처방"
  - "임상 검증", "임상 시험"
  - "100% 안전", "완전히 안전"
  - "약", "약을 드세요", "복용하세요"
  - "진단", "진단 결과"

허용 표현:
  - "~에 도움이 될 수 있어요"
  - "~를 고려해볼 수 있어요"
  - "참고 정보예요"
  - "전문의와 상담해주세요"
```

### 4.2 스캔 스크립트

```bash
# scripts/check-forbidden-expressions.sh
FORBIDDEN=(
  "치료합니다" "치료해요" "효과가 있습니다"
  "의사가 추천" "피부과 전용" "임상 검증"
  "100% 안전" "진단 결과"
)

for expr in "${FORBIDDEN[@]}"; do
  results=$(grep -rn "$expr" apps/web/ apps/mobile/ --include="*.tsx" --include="*.ts")
  if [ -n "$results" ]; then
    echo "⚠️ 금지 표현 발견: $expr"
    echo "$results"
  fi
done
```

## 6. 개인정보보호위원회 신고

### 6.1 신고 의무 기준

```
신고 필요:
  - 1,000명 이상 개인정보 유출
  - 민감정보 (건강정보) 유출 (건수 무관)
  - 고유식별정보 유출 (건수 무관)

신고 기한:
  - 인지 시점으로부터 72시간 이내
  - 개인정보보호위원회 (privacy.go.kr)
```

### 6.2 신고 내용

```
필수 기재:
  1. 유출 개인정보 항목
  2. 유출 시점 및 경위
  3. 이용자 피해 최소화 조치
  4. 이용자 대응 조치 (상담 창구 등)
  5. 담당 부서 및 연락처
```

### 6.3 사용자 통지 템플릿

```

■ 유출 항목: [항목 나열]
■ 유출 시점: [시점]
■ 유출 경위: [간단한 경위]
■ 조치 사항:
  - [취한 조치 1]

추가 문의 사항은 아래 연락처로 연락해주세요.
```

## 7. 암호화 키 관리

### 7.1 키 로테이션

```
SAFETY_ENCRYPTION_KEY 관리:
  - 저장: Vercel 환경변수 (서버사이드 전용)
  - 로테이션: 분기 1회 (또는 사고 발생 시 즉시)
  - 로테이션 절차:
    1. 신규 키 생성
    2. 기존 데이터 복호화 → 신규 키로 재암호화
    3. 환경변수 업데이트
    4. 배포 + 검증
    5. 기존 키 폐기
```

### 7.2 키 유출 대응

```
키 유출 발견 시:
  1. 즉시: Safety API 일시 중지
  2. 15분 내: 신규 키 생성 + 환경변수 교체
  3. 1시간 내: 전체 데이터 신규 키로 재암호화
  4. P1 사고 대응 플로우 병행
```
