# 리서치: "나 프로필" 통합 카드 — Phase 1.5

> **티켓**: ADR-098-R1
> **작성일**: 2026-04-26
> **상태**: 리서치 완료 → SDD-PROFILE-CARD-PHASE-1.5로 이관
> **목표**: 5축(PC/S/C/H/M) 결과를 홈 상단에 한 카드로 통합 노출하는 설계 근거
> **연관 ADR**: [ADR-098](../adr/ADR-098-identity-redefinition-5axis-model.md) §6 R1

---

## 1. 배경

ADR-098 §0 (P1 100점 기준):

> 5축이 하나의 프로필로 통합된 **"나 프로필"** 카드가 홈 상단 상시 노출.
> 사용자가 "이룸은 무슨 앱이야?"에 한 문장으로 답할 수 있음.

현재 상태(2026-04-26):

- 통합 분석 결과 페이지(`/analysis/integrated/result/[id]`)는 5축을 모두
  포함하지만 **결과 페이지 내부**에서만 노출됨
- 홈 최상단에는 [`IntegratedSessionPromptCard`](<../../apps/web/app/(main)/home/_components/IntegratedSessionPromptCard.tsx>)가
  있지만 이는 **"진입/재방문 유도 카드"** — 결과 자체를 압축 노출하지 않음
- 사용자는 매번 결과 페이지로 이동해야 자기 5축을 확인 가능 → 정체성이
  "상시 인지"되지 못함

Phase 1.5 목표: 통합 분석 완료자에게는 결과 압축 카드를, 미완료자에게는
기존 진입 유도 카드를 노출 — 상태 기반 분기.

## 2. 리서치 질문 (ADR-098 §6에서)

### Q1. 5축 결과를 하나의 카드로 통합했을 때 정보 밀도 한계는?

#### 분석

각 축이 가진 정보량 (PC-1·S-1·C-1·H-1·M-1):

| 축   | 핵심 출력                                | 추가 디테일                     |
| ---- | ---------------------------------------- | ------------------------------- |
| PC-1 | 시즌(Spring/Summer/Autumn/Winter) + 톤   | 베스트 5색, 워스트 5색, 립 추천 |
| S-1  | 타입(Dry/Oily/Combination/Normal/Sens.)  | 6존 점수, 신뢰도, 추천 루틴     |
| C-1  | 3타입(S/W/N)                             | 측정값 4종, 스타일링 원칙 4개   |
| H-1  | 얼굴형 + 모발 타입/굵기                  | 추천 헤어컷, 케어 루틴          |
| M-1  | 무드 키워드 (PC+S 조합 기반 실행 레이어) | 데일리/오피스/데이트별 분기     |

**카드 한 장 = 모바일 ~360pt 높이** (스크롤 없이 한눈에). 정보 밀도 제약:

- 타이포그래피 5축 × 라벨+값 = 10줄 → 화면 절반 점유
- 각 축에 신뢰도 시각화 추가 시 + 5요소
- 공유/더보기 액션 + 24pt 헤더 — 총 ~340pt

**결론**: 한 카드에 들어갈 수 있는 정보는 5축당 **2~3 데이터 포인트** 한도.
"핵심 라벨 + 신뢰도 + 한 줄 정체성 문장"이 상한.

#### 권장 정보 구조

```
┌─────────────────────────────────────────┐
│ [그라디언트 헤더]                        │  ← 24pt
│   봄 라이트 · 미니멀 스타일러            │  ← 한 줄 정체성 (18pt 볼드)
│   공유 ↗  더보기 →                       │  ← 12pt
├─────────────────────────────────────────┤
│   PC      S       C        H      M     │  ← 라벨 (10pt)
│   봄라이트 건성   스트레이트 계란형 내추럴 │  ← 값 (12pt)
│   ●●●●●  ●●●○○  ●●●●○   ●●●●●  ─      │  ← 신뢰도 (8pt 점)
└─────────────────────────────────────────┘
```

총 5칸 그리드 + 헤더 = ~280pt. 여유 60pt는 carousel 미니어처/페르소나
이미지 등 확장 슬롯.

### Q2. 축별 신뢰도가 다를 때 통합 카드에서 어떻게 처리?

#### 신뢰도 레벨 정의 (통합 분석 API의 `confidence`/`usedFallback` 기반)

| 레벨   | 점수              | 시각화                    | UX 반응                        |
| ------ | ----------------- | ------------------------- | ------------------------------ |
| 높음   | 85+               | ●●●●● (5점)               | 그대로 노출                    |
| 보통   | 70+               | ●●●●○ (4점)               | 그대로 노출                    |
| 낮음   | 60+               | ●●●○○ (3점)               | 그대로 + "재측정 권장" 칩 옵션 |
| 매우낮 | 60- 또는 fallback | ●●○○○ (2점) + 점선 테두리 | 값 노출하되 회색 + 재측정 CTA  |

#### 처리 원칙

1. **숨기지 않음** — 신뢰도 낮다고 축을 빼면 5축 통합 의미가 깨짐
2. **시각적 약화** — 점선 테두리 + 회색 톤으로 "참고 수준" 표시
3. **재측정 경로 제공** — 카드 내 모든 축 "탭하면 해당 분석 재진입" 가능
4. **신뢰도 평균이 60% 미만** → 카드 전체에 "재측정으로 정확도 향상" 배너

#### 데이터 소스

- 각 축의 `confidence` 필드 (통합 분석 결과 합성 시점에 저장됨)
- `usedFallback` true → 신뢰도 매우낮으로 강제 분류
- M-1은 PC+S 조합 결과의 파생 — 자체 신뢰도 없음, 표시는 ─ 또는 비워둠

### Q3. 사용자가 통합 프로필을 공유하고 싶어할 때 UX 패턴은?

#### 공유 시나리오

| 시나리오                 | 매체                          | 익명 처리       |
| ------------------------ | ----------------------------- | --------------- |
| 친구에게 "내 결과 봐"    | 카카오톡 / 인스타 DM (이미지) | 닉네임 노출     |
| SNS 자랑 (인스타 스토리) | 인스타그램 (이미지)           | 익명 옵션 권장  |
| 블로그/리뷰              | 링크 (OG 카드)                | 자동 익명 (URL) |
| 캡처 후 직접 공유        | 스크린샷 (이미지)             | 사용자 책임     |

#### 공유 패턴 권장

1. **카드 → PNG 변환 옵션 우선** — `html-to-image` 라이브러리 (기존 분석
   결과 공유에서 검증된 패턴 — `useAnalysisShare` 훅 재활용)
2. **익명 모드 토글** — 닉네임/프로필 사진 숨김 + "이룸 사용자" 표시
3. **링크 공유는 Dynamic OG** — `/analysis/integrated/share/[id]` 페이지에
   메타 OG 카드 자동 생성 (기존 출시 인프라 ADR-072 참조)
4. **공유 카운트 기록** — `integrated_analysis_sessions.share_count`로
   가벼운 어뷰즈 방지 (1일 5회)

#### 카드 PNG 사양

- 정사각(1080×1080) + 세로(1080×1920) 두 포맷 (인스타 피드/스토리)
- 상단: 5축 칩 그리드
- 중앙: 한 줄 정체성 문장 (그라디언트)
- 하단: 이룸 로고 + URL `yiroom.app`

## 3. 한 줄 정체성 문장 생성 알고리즘

### 입력

```typescript
interface IntegratedProfileInput {
  pc: { season: 'Spring' | 'Summer' | 'Autumn' | 'Winter'; tone: string };
  skin: { type: SkinType };
  body: { type: 'S' | 'W' | 'N' };
  hair: { faceShape: FaceShape; type: HairType };
  makeup: { mood: MakeupMood }; // PC+S 조합으로 도출된 키워드
  confidence: { pc: number; skin: number; body: number; hair: number };
}
```

### 알고리즘 — 2-슬롯 템플릿

```
"{핵심 색-체형 키}의 {스타일 무드} 스타일러"

예시:
- 봄 라이트 + S 체형 → "봄 라이트의 미니멀 스타일러"
- 가을 딥 + W 체형 → "가을 딥의 클래식 스타일러"
- 여름 뮤트 + N 체형 → "여름 뮤트의 캐주얼 스타일러"
- 겨울 트루 + S 체형 → "겨울 트루의 모던 스타일러"
```

### 무드 매핑 테이블 (체형 → 무드)

| 체형 | 무드 키워드                |
| ---- | -------------------------- |
| S    | 미니멀 / 모던 / 클린       |
| W    | 페미닌 / 클래식 / 로맨틱   |
| N    | 캐주얼 / 내추럴 / 보헤미안 |

### 무드 변형 (시즌 톤별)

| 시즌 톤 | 추가 무드 형용사 |
| ------- | ---------------- |
| Light   | 미니멀, 클린     |
| Bright  | 발랄, 비비드     |
| True    | 클래식, 모던     |
| Mute    | 잔잔, 빈티지     |
| Deep    | 시크, 강렬       |

### 결정 로직

```typescript
function generateIdentitySentence(input: IntegratedProfileInput): string {
  const { pc, body } = input;
  const seasonLabel = `${seasonKo(pc.season)} ${toneKo(pc.tone)}`;
  const moodPool = MOOD_BY_BODY[body.type];
  const seasonAdj = MOOD_BY_TONE[pc.tone];
  const mood = pickMood(moodPool, seasonAdj); // 톤 형용사가 풀에 있으면 우선
  return `${seasonLabel}의 ${mood} 스타일러`;
}
```

### 신뢰도 낮을 때 fallback

- PC 신뢰도 60% 미만: `"내 시각 정체성을 만들어가는 중"` (분석 진행 권유)
- Body 신뢰도 60% 미만: `"{seasonLabel}의 스타일러"` (체형 무드 생략)
- 둘 다 60% 미만: `"이룸과 함께 시작하기"` (일반 카피)

## 5. 결론 (Phase 1.5 SDD 전제)

1. **카드 구조**: 헤더(한 줄 문장) + 5축 칩 그리드 + 신뢰도 점 + 액션 2개
2. **신뢰도 분기**: 4단계 시각 약화 + 재측정 CTA, 숨김 없음
3. **공유**: PNG 변환 + 익명 모드 + Dynamic OG 링크
4. **카피 알고리즘**: 시즌·톤·체형 3축 결합 → 단순 템플릿. M-1은 카피
   생성에 직접 영향 X (PC+S 파생이라 중복)
5. **상태 분기**: 통합 세션 있음 → 압축 카드, 없음 → 기존 진입 카드
6. **출시 후 4~8주 내 도입 가능** — 신규 API 불필요 (기존 통합 분석 결과
   재사용), `FEATURE_FLAGS.PROFILE_CARD_PHASE_1_5` 플래그로 제어
