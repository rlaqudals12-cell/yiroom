## 과제 A — 현행 이룸 IA 실측 (apps/web)

이룸 웹은 **두 개의 완전히 다른 내비게이션 멘탈 모델**을 하나의 제품에 얹고 있다.

- **데스크톱**: `components/Navbar.tsx` — 상단 헤더 + **드롭다운 중심** (기능을 카테고리 폴더로 숨김)

### A-4. "내 정보 보기" 표면 — 중복·불명확 지점

가장 심각한 문제. 사용자의 "내 정보/내 결과"가 **3개 표면에 분산·중복**되어 있다.

| --------------- | ------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| `/home`         | 3-State 홈(New/Growing/Active) + **`ProfileCardGrid` (5축 채워지는 프로필, ADR-111)**                               | 홈인데 프로필 카드가 정본으로 박혀 있음                                              |
| `/dashboard`    | **`/home`으로 리다이렉트** (빈 껍데기)                                                                              | 데스크톱 Navbar가 "대시보드"·"내 분석 결과" **두 라벨**으로 이 리다이렉트를 가리킴   |

| 지표                                 | 현황                                    |
| ------------------------------------ | --------------------------------------- |
| 데스크톱 최상위 진입점(노출)         | 6개 + 분석 드롭다운 하위 7개            |
| 데스크톱 잠재 진입점(게이팅 해제 시) | 10개 + 하위 17개                        |
| 모바일 하단 탭(노출)                 | 4개                                     |
| `(main)` 최상위 세그먼트             | 39개                                    |
| `page.tsx` 총 개수                   | 141개                                   |
| "내 정보" 정본 표면                  | **3개 중복** (home·dashboard·profile)   |

**3대 핵심 문제**

1. **이중 IA**: 데스크톱(드롭다운) ↔ 모바일(탭)이 다른 멘탈 모델 — 학습 비용 2배.
2. **정본 중복**: 5축 프로필이 홈·나에 이중 존재, 홈·대시보드·내분석결과가 같은 곳을 가리킴.
3. **컨설팅의 부재**: 재편 방향의 심장인 "물어보기(AI 코치)"가 네비에 **없다**. 브리핑을 줄 홈은 있으나, 각 기능이 "따로따로"라 하나의 컨설팅 흐름이 아니다.

## 과제 B — 웹 리서치 (2025-2026)

### B-1. 모바일 앱 IA 표준 — 하단 탭 3~5개

세 권위 기관이 모두 **3~5개**로 수렴한다.

| 기관                                   | 규칙                    | 근거                                                                                                                                                                                        |
| -------------------------------------- | ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Apple HIG** (Tab bars)               | iPhone에서 **탭 3~5개** | "탐색에 필요한 **최소 개수의 탭**을 사용하라. 탭이 하나 늘 때마다 앱의 복잡성이 커져 정보 찾기가 어려워진다." **"More(오버플로) 탭을 피하라"** — 콘텐츠를 숨기고 도달·인지를 어렵게 만든다. |
| **Material Design 3** (Navigation bar) | **목적지 3~5개**        | "**목적지가 3개 미만이면 내비게이션 바를 쓰지 말고 탭을 써라.** 5개를 넘기지 말라 — **탭 타깃이 너무 가까워진다.**" 5개 초과 시 modal expanded rail 권장.                                   |
| **Nielsen Norman Group**               | 실질 상한 **5**         | "옵션이 5개를 넘으면 **적정 터치 타깃 크기를 유지하며 탭바에 담기 어렵다.** 숨겨진 옵션은 **'눈에서 멀어지면 마음에서도 멀어진다'** — 사용자는 스크롤할 생각을 안 한다."                    |

**5개가 상한인 두 독립적 이유**: (1) 물리적 — 터치 타깃 과밀, (2) 인지적 — 탭마다 멘탈 모델이 희석되고 More 탭이 콘텐츠를 매장. → 이룸의 "진입점 과다"는 교과서적 위반. **하단 탭 3~5개, 이상적으로 4개**, More 오버플로 금지.

**허브 앤 스포크 vs 탭바 (NN/g 정의)**

- **탭바**: 스크롤과 무관하게 **항상 보이는** 지속 바. **소수의 비계층적 목적지**, "한 화면이 다른 화면보다 특별히 중요하지 않고 자유롭게 오가는" 앱에 적합. 소셜·커머스. 장점: 항상 접근·암기 불필요(다양한 숙련도 사용자에 이상적). 단점: ~5개 상한.
- **허브 앤 스포크**: 중앙 허브가 **모든 옵션을 나열**, 섹션 전환 시 **허브로 복귀**. **태스크 기반**, "한 세션에 한 브랜치만 쓰는" 앱에 적합. 단점: "**홈의 노른자 공간을 메뉴(chrome)에 낭비**", 전환마다 한 단계 추가.

**컨설팅/브리핑 앱에 맞는 구조 = 하이브리드**: 순수 허브(아이콘 런처)가 아니라 **"콘텐츠가 곧 홈"인 탭바**. 홈 탭 = 브리핑 자체(메뉴 아님) + 나머지 잡(job)을 위한 2~4개 형제 탭. → **≤5개면 탭바, 그 이상 넘칠 때만 허브**. 이룸의 해법은 "메뉴 버튼을 늘리는 것"이 아니라 **기능을 소수 탭 + 하나의 깊은 주 표면으로 접는 것**.

### B-2. 브리핑형 홈 사례 (매일 여는 앱)

| 앱         | 홈 설계 핵심                                                                                                                                                                                                                     |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **WHOOP**  | 수십 개 생체신호를 **회복 점수 하나(0~100)**로 압축. 오버뷰는 딱 3지표(Recovery/Strain/Sleep). 주 지표는 **~72pt 크게** 렌더, 보조 텍스트는 작게. **3색(초록=준비됨/빨강=위험/노랑=중간)** 만 반복. 아침 다크 UI 의례.           |
| **Oura**   | 복잡한 생체데이터를 **3점수(Readiness/Sleep/Activity)** 로 종합, **아침 요약**이 "오늘 얼마나 무리해도 되는지"를 지시. 철학: 사용자는 **"데이터가 아니라 의미를 듣고 싶다"** (= AI 컨설턴트 테제). 앱은 **Today** 표면으로 열림. |
| **Garmin** | "Morning Report" — HRV·수면·날씨를 **한 장의 아침 브리핑**으로.                                                                                                                                                                  |

**공통 패턴 (매일 여는 브리핑 홈)**

1. 앱이 **메뉴가 아니라 콘텐츠로 직접 열림** (Today/Home = 리포트 그 자체)
2. 상단에 **결론 하나(점수/상태/헤드라인)** — 여러 입력을 압축
3. **점진적 공개**: 상세는 한 탭 아래, 표면은 차분하게
4. **일관된 최소 의미색 + 큰 타이포**
5. 반복되는 **아침 의례** 설계

→ 이룸 직접 대응: 아침 브리핑 = **결론 헤드라인 하나**("오늘의 컨디션/스타일 브리핑"), 5축 정체성을 한눈 상태로 압축, "왜?"·전체 분석은 점진 공개. 기존 `computeSkinTrend`("오늘의 컨디션") 개념과 정합.

**토스(Toss) IA 단순화 철학** — 한국 10~30대의 기대치라 특히 중요:

- **"1 Thing / 1 Page"**: 토스 UI의 최대 특징 = **페이지마다 한 가지 정보/한 가지 행동** (가입 시 화면당 입력 1개). "one screen, one job"의 구체형.
- **홈 재설계의 무게**: "1,000만 사용자가 만족할 홈"을 위한 전담 노력. 홈은 "너무 많은 사용자가 의존해 **바꾸기가 특히 어렵다**"(Simplicity 컨퍼런스).
- **디자인 시스템(TDS)** 으로 "반복적 UI 결정"을 제거 → 방대한 기능을 하나의 일관된 단순 제품으로.
- ⚠️ 솔직성 플래그: 토스 홈 개편도 **"불편하다" 사용자 반발**(Blind)을 겪음 — 매일 여는 홈은 최고 난이도·논쟁적 표면이므로 반드시 사용자 테스트.

| 앱                     | 하단 탭 (신뢰도)                                                                                                               | 주 탭                                                          |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------- |
| **화해 (Hwahae)**      | 약 5개 — **홈 / 랭킹 / 쇼핑 / 카테고리 / MY(마이)** _(중간 신뢰도; MY·랭킹·쇼핑 표면은 공식 문서화, 정확한 5라벨 바는 미검증)_ | 홈(랭킹·"내 피부 맞춤 랭킹템")                                 |
| **잼페이스 (Jamface)** | **미검증** — 신뢰 가능한 공개 소스 없음. 진단(퍼컬·스킨)·매칭 기능은 확인됨. 라이브 앱 직접 확인 권장                          | 사진 1장 진단 우선 흐름 (이룸 5축 캡처와 가장 구조적으로 유사) |
| **에이블리 (Ably)**    | **5~6개** — 홈 / 카테고리 / 혜택 / 콘텐츠 / 찜 / 마이페이지 _(과거: 홈/마켓/샥출발/찜/마이페이지)_ _(중간 신뢰도)_             | 홈(AI 개인화 상품 피드)                                        |

**한국 사용자가 기대하는 패턴**

- **홈 탭 = 개인화 콘텐츠로 열림**(런처 아님)
- **마이(MY/마이페이지)** = 최우측 표준 슬롯
- 중간에 카테고리/랭킹/혜택
- 커머스 앱은 5~6탭까지 늘리지만(머천다이징 breadth 우선), **콘텐츠/컨설팅 앱은 4탭**으로 차분하게 — 목표가 다름.

### B-4. "기능 많은 앱이 단순하게 느껴지게" 하는 IA 원칙

1. **최상위 진입점 최소화**: Apple "최소 개수의 탭". NN/g "최상위 카테고리가 많을수록 사용자가 평가하기 더 힘들다." → 지름길을 늘리지 말고, **의미가 풍부한 소수의 선택지**로.
2. **깊이(depth)는 적이 아니다 — "3클릭 룰"은 신화**: NN/g "3클릭 룰은 데이터로 뒷받침되지 않는 임의 규칙", "클릭 수 세기 자체는 무의미한 지표." 중요한 건 **정보 향(scent)·명확한 길찾기·단계별 낮은 인지 부담**. → **전체 분석을 더 깊이(점진 공개) 둬도 안전**, 라벨만 분명하면 됨.
3. **강한 정보 향(information scent) 라벨**: "모호하거나 브랜드화된 낯선 용어 회피." → 탭·섹션은 **평이하고 의미 있는 단어**("오늘", "내 프로필", "옷장", "제품"), 향 없는 브랜드 조어 금지.
4. **점진적 공개(progressive disclosure)** — 여기서 가장 중요한 도구: "고급·희귀 기능을 2차 화면으로 미뤄 학습성↑·효율↑·오류↓, **기능은 제거하지 않음.**" **실패 모드 경고**: "자주 쓰는 것은 반드시 앞에 노출하고 1차 목록을 작게 유지해야 한다 — 아니면 복잡성을 **줄이는 게 아니라 이동**시킬 뿐." → 창업자 방향 "결론 먼저·근거는 접기"와 정확히 일치.
5. **검색/명령 진입 = 압력 밸브**: 화해·에이블리·토스 모두 홈 상단에 검색. 파워 유저가 바로 점프 → 탭바를 4~5개로 유지. 단, 좋은 IA의 **대체가 아니라 보완**.
6. **압축 + 결론 하나**: WHOOP/Oura 교훈 — 여러 입력을 **한 눈 점수/결론**으로 압축, 크게, 좁은 의미색. 데이터 밀도 높은 제품이 "단순하게 느껴지는" 방법.
7. **시스템 레벨 일관성**: 토스/TDS 교훈 — 디자인 시스템이 "반복 UI 결정"을 없애 방대한 기능을 하나의 일관된 단순 제품으로.

## 이룸 제안 탭 구조 (2~3안)

전 안 공통 원칙 (리서치 도출):

- **하단 탭 = 데스크톱·모바일 통일** (이중 IA 폐기). 데스크톱은 같은 탭을 사이드/상단으로 표현하되 **동일 라벨·동일 계층**.
- **홈 = 아침 브리핑 콘텐츠**(런처 아님). 결론 헤드라인 하나 + 점진 공개.
- **"내 정보" 정본 1곳**으로 통합 (`ProfileCardGrid` 중복 제거).
- **평이한 라벨**, More 오버플로 금지, 최우측 = 마이.
- 나머지 39개 세그먼트는 **탭 하위로 흡수**(점진 공개) — 삭제가 아니라 재배치.

---

### 안 1 — 3탭 (극단적 단순, 컨설팅 순수형)

**[오늘] · [물어보기] · [내 정보]**

가장 토스/Oura적. "컨설팅 = 대화"를 1급 탭으로 승격.

| 탭                       | 역할                                                                       | 흡수하는 현행 기능                                                                                                    |
| ------------------------ | -------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **오늘** (`/home`)       | 아침 브리핑. 오늘의 컨디션·스타일 결론 하나 + 오늘의 추천(제품/코디) + TPO | home 3-State, 오늘의 추천, 캡슐(Daily Capsule), 날씨 코디, beauty·style 피드의 "오늘" 슬라이스                        |
| **물어보기** (`/coach`)  | AI 컨설턴트 대화(사진 판정 포함). "색→옷→피부→제품" 하나의 흐름            | coach, chat, 각 분석 결과의 상담 CTA, 통합분석 진입(대화로 시작)                                                      |
| **내 정보** (`/profile`) | 5축 정체성 정본 + 옷장 + 제품/위시 + 설정                                  | profile(5축 ProfileCardGrid 정본 **단일화**), closet/outfit/styling, products/wishlist, 분석 히스토리, 소셜, settings |

- **장점**: 진입점 최소, 컨설팅 정체성 최강, "따로따로" 완전 해소.
- **위험**: "내 정보"가 과부하(옷장·제품·소셜·설정) → 점진 공개 설계 난이도 높음. 커머스(제품/어필리에이트) 표면이 깊이 묻혀 수익 표면 약화.

---

### 안 2 — 4탭 (균형, **추천안**)

**[오늘] · [물어보기] · [스타일] · [내 정보]**

리서치 권고(4탭)·한국 관습(최우측 마이)·컨설팅 정체성·수익 표면을 동시 충족.

| 탭                       | 역할                                                                   | 흡수하는 현행 기능                                                                                                              |
| ------------------------ | ---------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| **오늘** (`/home`)       | 아침 브리핑 홈. **결론 헤드라인 하나** + 오늘의 추천 + 미션. 메뉴 아님 | home 3-State, 오늘의 추천, Daily Capsule, 날씨 코디, WelcomeBack                                                                |
| **물어보기** (`/coach`)  | AI 스타일리스트·피부관리사 대화. 사진 판정, 통합 컨설팅 흐름의 관문    | coach, chat, 통합분석 시작, 각 결과 상담 CTA, 시술/루틴 추천                                                                    |
| **스타일** (`/style`)    | 옷장·코디·제품 쇼핑(커머스/어필리에이트 표면). "무엇을 살까/입을까"    | beauty(제품 피드), style/styling/outfit/saved-outfits, closet, products, wishlist, capsule 쇼핑, VTO                            |
| **내 정보** (`/profile`) | **5축 정체성 정본 1곳** + 분석 히스토리 + 설정                         | profile(ProfileCardGrid **정본 단일화**), 5축 결과·재분석, Before/After, 등급, 소셜(친구/리더보드/챌린지), settings/공지/도움말 |

- **`/dashboard`·"내 분석결과"** → `/profile`로 일원화(리다이렉트 라벨 정리).
- **홈의 `ProfileCardGrid` 제거** → 홈은 브리핑만, 프로필 정본은 "내 정보"에만.
- **장점**: 4탭 리서치 정합 · 컨설팅("물어보기") 1급 승격 · 수익("스타일") 독립 표면 · 마이 최우측.
- **왜 추천**: 창업자 재편 3요소(브리핑 홈 · AI 컨설턴트 · 가입=통합분석)를 탭에 1:1 매핑하면서, 제품/어필리에이트(P5 수익)를 묻지 않는 유일한 안.

---

### 안 3 — 4탭 (현행 최소 변경, 뷰티앱 관습형)

**[홈] · [뷰티] · [스타일] · [내 정보]** (현행 유지 + 코치 통합)

현행 BottomNav를 거의 보존하되, 누락된 "물어보기"를 **각 탭 상단 플로팅 진입/홈 브리핑 카드**로 통합.

| 탭                    | 변경                                             |
| --------------------- | ------------------------------------------------ |
| **홈**                | 브리핑화(결론 헤드라인) + `ProfileCardGrid` 제거 |
| **뷰티**              | 유지 (피부 기반 제품 피드)                       |
| **스타일**            | 유지 (체형 기반 코디 피드)                       |
| **내 정보**           | 5축 정본 단일화 + 소셜/설정                      |
| (플로팅) **물어보기** | 모든 탭 상단 우측 AI 코치 진입 버튼 (탭 아님)    |

- **장점**: 현행 코드 변경 최소, 뷰티앱(화해/에이블리) 관습에 가장 근접, 마이그레이션 리스크 낮음.
- **위험**: "물어보기"가 탭이 아닌 플로팅 → 컨설팅 정체성이 여전히 2급. "뷰티 vs 스타일" 분리가 사용자에겐 여전히 "따로따로"로 느껴질 수 있음(피부는 뷰티? 제품은?).

### 안 비교 요약

| 기준                             | 안 1 (3탭) | **안 2 (4탭·추천)** | 안 3 (4탭·현행형)        |
| -------------------------------- | ---------- | ------------------- | ------------------------ |
| 리서치 탭 수 정합(3~5, 4 이상적) | ✅(3)      | ✅(4)               | ✅(4)                    |
| 컨설팅("물어보기") 1급화         | ✅         | ✅                  | ⚠️ 플로팅                |
| 수익(제품/어필리에이트) 표면     | ⚠️ 묻힘    | ✅ 독립             | ✅ 뷰티+스타일           |
| "따로따로" 해소                  | ✅ 최강    | ✅                  | ⚠️ 뷰티/스타일 분리 잔존 |
| 마이그레이션 비용                | 높음       | 중간                | 낮음                     |
| 한국 뷰티앱 관습 근접            | ⚠️         | ✅                  | ✅ 최강                  |

**권고**: **안 2**를 기준으로 하되, 마이그레이션 리스크가 크면 **안 3에서 안 2로 단계 이행**(뷰티+스타일 → 스타일 통합, 플로팅 코치 → 탭 승격)을 로드맵화. 어느 안이든 **① 이중 IA 폐기 ② 홈=브리핑화 ③ 프로필 정본 단일화**는 공통 선행 과제.
