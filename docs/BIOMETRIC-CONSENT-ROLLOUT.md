# 생체동의 게이트 롤아웃 & 글로벌/미국 홍보 준비 런북

> 작성 2026-07-12. 목적: 미국·글로벌 커뮤니티 홍보를 **법적으로 안전하게** 열기 위한 순서.
> 핵심 명제: **미국 시장은 포기 대상이 아니다. 생체동의 게이트 배포가 곧 미국 합법 진입이다.**

---

## 1. 왜 지역 차단이 아니라 동의 게이트인가

- **BIPA(일리노이)**는 생체정보 처리를 *금지*하지 않는다. 요구하는 것은 3가지:
  1. 수집 사실·목적·보유기간을 **서면 고지**
  2. 수집 **전** 서면 **동의(release)**
  3. **공개된 보유·파기 정책**
- 이룸은 이 3가지를 이미 갖출 수 있다: 동의 게이트(`requireBiometricConsent`) + 처리방침(§4·§5 국외이전) + 파기 정책(1년/삭제 시, B3에서 삭제 완전화 완료).
- 따라서 **게이트를 켜면 일리노이 포함 미국 전역에서 합법**. 지역 차단은 확장성을 죽이는 잘못된 선택.

## 2. 게이트 현황 (점검 완료)

- 코드 = 프로덕션 품질. `wip/biometric-consent-DO-NOT-DEPLOY` 브랜치에 코어 존재:
  - `lib/api/biometric-consent.ts` — fail-closed 게이트(미동의·컬럼없음·조회실패 시 403)
  - `supabase/migrations/20260712_biometric_consent.sql` — `user_agreements`에 `biometric_agreed/at/version` 컬럼(IF NOT EXISTS·롤백 포함)
  - `components/agreement/types.ts` — 생체 동의 항목(필수, Gemini 미국 전송 명시)
  - `app/api/agreement/route.ts` · `AgreementGuard.tsx` · `app/agreement/page.tsx` — 동의 저장·검증·게이트
  - 7개 분석 라우트에 `requireBiometricConsent` 배선
- **남은 코드 작업 1건**: wip 브랜치 분석 라우트는 _연령 게이트 배선 전_ 버전 → 현재 워킹트리(연령 게이트 있음)와 합칠 때 7개 라우트에 `requireBiometricConsent`를 `requireAgeVerified` **옆에** 추가하는 소규모 병합.
- **주의**: wip 브랜치 커밋에는 어필리에이트 고지 배선도 섞여 있으나, 그건 B1이 워킹트리에 새로(더 낫게) 배선했으므로 **가져오지 않는다**. 생체 코어 파일만.

## 3. ⚠️ 롤아웃 순서 (반드시 준수 — 순서 틀리면 전 사용자 403)

게이트는 fail-closed다. prod에 컬럼이 없는 상태로 배포하면 조회 에러 → 모든 분석 403. push=자동배포이므로 특히 주의.

1. **[prod] 마이그레이션 먼저** — Supabase 대시보드 SQL Editor에서 `20260712_biometric_consent.sql` **수동 실행**(파괴적 `db push` 금지). 컬럼 생성 확인.
2. **[코드] 게이트 병합·배포** — 생체 코어 + 7라우트 연령게이트 공존 병합 → 커밋 → push.
3. **[사용자] 기존 사용자 재동의** — 기존 가입자는 `biometric_agreed=false` → `AgreementGuard`가 `/agreement`로 유도 → 생체 동의 체크 후 진행. dev <100명이라 수용 가능.
4. **[검증]** 배포 후: 동의 없이 `/api/analyze/skin` 직접 호출 → 403 확인. 동의 후 분석 성공 확인.

## 5. 권장 실행안

1. **지금**: A/B그룹을 커밋·배포·안정화(fail-closed 아님, 안전). 웹 라이브 즉시 개선.
2. **미국 홍보 직전**: 생체 게이트를 §3 순서로 배포(마이그 먼저). 이때 §4 문구도 정합.
3. **병행**: 도메인 구매 → pk_live 전환(실런칭 품질). 이게 미국/글로벌 홍보의 실질 전제.
4. 게이트 코드 병합은 A/B 배포 후 착수하면 분석 라우트가 연령게이트를 base로 가져 병합이 깔끔.
