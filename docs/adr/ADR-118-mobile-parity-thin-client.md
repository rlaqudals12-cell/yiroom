# ADR-118: 모바일 패리티 공유 방식 — 웹 API 정본 + Thin Client

- **상태**: Accepted
- **날짜**: 2026-07-10
- **관련**: ADR-102(통합분석 모바일 API), ADR-114(전속 뷰티팀 IA), ADR-117(맞춤 루틴 엔진)

## 맥락

모바일 앱 패리티 5~7단계(루틴 엔진·브리핑·AI 트윈)는 웹의 대형 도메인 로직을 모바일에서도 써야 한다. 후보는 세 가지였다:

1. **복붙 포팅** — 웹 lib를 모바일 lib로 복사 (1~4단계에서 부분 사용)
2. **packages/shared 승격** — 순수 로직을 공유 패키지로 이동, 양앱 import
3. **웹 API 정본 + thin client** — 웹에 API 라우트 신설, 모바일은 fetch만

## 결정: 3안 (웹 API 정본 + thin client)

| 기준          | 복붙 포팅        | shared 승격                     | **API 정본**                          |
| ------------- | ---------------- | ------------------------------- | ------------------------------------- |
| 드리프트 위험 | 높음(두 벌 유지) | 없음                            | **없음(정본 1곳)**                    |
| 웹 회귀 위험  | 없음             | **높음**(import 경로 대량 변경) | 없음(라우트 추가만)                   |
| 모바일 공수   | 중               | 중                              | **소(fetch+UI)**                      |
| 오프라인      | 가능             | 가능                            | 캐시 폴백(AsyncStorage TTL)           |
| 전례          | 스캔 verdict(M3) | 없음                            | **통합분석(ADR-102)·outfits·workout** |

- 브리핑(`composeBriefing`)·루틴(`generateRoutine`+step-spec+cycling)은 서버 저장 정본 데이터의 조립이라 API가 자연스럽다. 온디바이스 원칙은 **카메라 이미지 분석**(업로드 비용·프라이버시)에만 적용 — 이원화가 아니라 역할 분리.
- 트윈은 이미 서버 생성(나노바나나)이라 애초에 API 호출이 정본.

## 적용 범위

- **5단계 브리핑**: 웹 `/api/briefing` 신설(composeBriefing 서버 조립) + 모바일 [오늘] 탭 fetch·캐시 폴백
- **6단계 루틴**: 웹 `/api/routine/daily` 신설 + 모바일 루틴 표면 fetch
- **7단계 트윈**: 기존 웹 트윈 API 호출 + 모바일 화면만
- 1~4단계에서 이미 포팅된 순수 로직(top-actions, scan verdict, 타임라인)은 유지 — 재공사하지 않는다(P4)

## shared 승격 재검토 조건

세 번째 소비 앱이 생기거나, 오프라인 우선 요구가 확정되거나, API 왕복이 실측 병목이 될 때. 그전까지는 승격하지 않는다(P0: "혹시 모르니까" 금지).

## 되돌림

Type 2(되돌릴 수 있음) — API 라우트는 유지한 채 shared 승격으로 전환 가능(클라이언트 교체만).
