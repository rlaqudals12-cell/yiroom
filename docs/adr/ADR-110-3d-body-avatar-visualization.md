## Context

로드맵 Phase 4(동반자)의 유일 잔여 = 3D 아바타. 2026-07-06 리서치 결론: 치수 추정(cm)이 아니라 **아바타 시각화**가 옳은 경로 (스케일 모호성 + ZOZOSUIT 기대-실망 갭 + 성공 사례의 셀링포인트 = "아바타 변화 시각화").

### 리서치 이후 새로 확인된 제약 2건

**① SMPL 에셋 라이선스 (리서치 문서 미포함, 2026-07-08 확인)**

SMPL 모델 파일(메시·형상 기저)은 **비상업 연구 전용** 라이선스다. 상업 제품 탑재·상업 서비스 사용이 명시적으로 금지되며, 상용은 Meshcapade 유료 라이선스가 필요하다. 이룸은 상용 앱이므로 **SMPL 에셋을 그대로 탑재하는 경로는 라이선스 위반**이다.

- 모델 라이선스: https://smpl.is.tue.mpg.de/modellicense.html
- 상용 창구: https://meshcapade.com/smpl/ (sales@meshcapade.com)

**② 프로드 데이터 실측 (2026-07-08, Management API 실쿼리)**

`body_analyses` 7행 검증 결과:

| 데이터                      | 가용성                                                                     |
| --------------------------- | -------------------------------------------------------------------------- |
| `body_type` (S/W/N)         | **7/7 전 행** — 유일하게 항상 있는 입력                                    |
| `ratio` (어깨/허리 비율)    | 2/7 — **최신 통합 경로 행만** (1.12, 1.15)                                 |
| `shoulder/waist/hip` 0-100  | 5/7 — 구형 V1 경로 행만. 최신 행엔 null                                    |
| `height/weight`             | 5/7 — 최신 통합 행엔 null                                                  |
| BodyRatios 전체 (11개 비율) | **0/7 — 미저장.** body-v2가 클라에서 측정하나 통합 경로는 ratio 1개만 저장 |

부수 발견 (유령 배선 #13, #14):

- **`body_assessments` 테이블은 프로드에 존재하지 않음** → 단독 `/api/analyze/body-v2` 라우트의 DB 저장은 프로드에서 항상 실패(합성 응답 폴백으로 은폐됨). 아바타는 이 테이블을 읽으면 안 됨. (수리는 본 ADR 범위 밖 — 별도 이슈)
- `measurement_source` 컬럼(20260619 마이그) 프로드 미적용 — best-effort 설계라 무해하나 gap-apply 대기 목록에 있음.

### 요구사항 (P0 검증)

- 왜? ① 체형 축의 "시각적 자기이해" 가치 완성 (숫자 없는 직관 전달) ② 재분석 변화를 보이게 (리서치: 숫자보다 변화) ③ 데모·차별화 자산 (프로필 홈의 시각 정점)
- 삭제 가능? 로드맵 Phase 4 잔여 항목으로 사용자 승인된 방향. 단, cm 수치 표시는 삭제(원칙).

## Decision

**절차적 파라메트릭 메시 (Procedural Parametric Mesh)** 를 채택한다. SMPL의 **원리**(형상 = 기본형 + 축별 선형 변형)는 차용하되, SMPL **에셋**은 사용하지 않는다.

### 핵심 결정 5가지

1. **메시 = 코드에서 절차 생성** — 높이별 타원 단면 로프팅 + 캡슐 사지. 에셋 파일 0KB, 라이선스 청정, 결정론. 모프 가중치가 어깨/허리/힙 제어점을 스케일.
2. **스타일 = 중립 저폴리곤 스타일라이즈드** — 얼굴 없음, 중립 톤, 실루엣·비율만 정확. photoreal 금지 (ZOZOSUIT 기대-실망 갭 완화가 목적이므로 시각 스타일 자체가 기대 수준을 설정).
3. **입력 = `body_analyses` 기존 컬럼만, 계층적 강등(tier)**:
   - Tier 0 (항상): `body_type` S/W/N → 타입 프리셋 모프
   - Tier 1 (있으면): `ratio` → 어깨/허리 모프 정밀화. 구형 행은 `shoulder/waist/hip` 0-100에서 근사 도출
   - Tier 2 (확정, 2026-07-08): **v1과 함께 저장 시작** — 통합 경로에 BodyRatios 전체 저장 (JSONB 1컬럼, 마이그 1건 + prod gap-apply). 데이터는 쌓여야 가치가 생기므로 일찍 시작. 아바타 v1 렌더는 여전히 type+ratio로 동작(저장과 소비를 분리)
4. **AI 호출 없음** — 아바타는 저장된 분석 결과의 **결정론적 파생 뷰** (순수 함수). Hybrid 데이터 패턴의 "표시 데이터" 계층이며 재현성 원칙(신뢰 1순위)을 상속.
5. **표면 = 웹 우선 2곳** — 체형 결과 페이지 + 프로필 체형 축. Three.js는 dynamic import + ssr:false + 뷰포트 지연 로드, WebGL 실패 시 기존 2D 실루엣 폴백. 재분석 오버레이 비교(고스트 중첩)는 v1에 포함하되 ratio 이력 2행 이상인 사용자에게만 노출.

## Alternatives (검토한 대안)

| 대안                                 | 판정        | 이유                                                                                                                           |
| ------------------------------------ | ----------- | ------------------------------------------------------------------------------------------------------------------------------ |
| A. SMPL 런타임 탑재                  | ❌ 기각     | 에셋 라이선스 = 비상업 전용, 상용은 Meshcapade 유료. 게다가 v1 요구(타입+비율 시각화)에 이미지 피팅·정밀 형상 기저는 과잉 (P4) |
| B. 사전 제작 GLTF + 블렌드셰이프     | △ 보류      | 라이선스 청정하나 3D 저작 파이프라인(Blender) 신설 필요, 에셋 용량(수백 KB~MB), 수정 비용. 절차 생성으로 충분하면 불필요       |
| C. **절차적 파라메트릭 메시** (채택) | ✅          | 에셋 0KB·라이선스 청정·결정론·코드만으로 튜닝. 리스크는 "예쁘게 만들기" 난이도 — 스타일라이즈드 방향이라 허용 가능             |
| D. 2D SVG 실루엣 모핑 (3D 아님)      | △ 폴백 채택 | 가장 싸지만 "회전 가능한 3D"의 데모·차별화 가치 미달. WebGL 폴백으로만 사용                                                    |
| E. Meshcapade 상용 라이선스 구매     | ❌ 기각     | 비용 대비 과잉 (v1은 이미지→형상 피팅 없음). 향후 사진 기반 피팅 도입 시 재검토                                                |

## Consequences

### 긍정

- 라이선스 리스크 0, 번들 추가분은 Three.js(~150KB gzip, 지연 로드)뿐
- 결정론 → 재현성 신뢰 전략과 정합, 테스트 용이 (정점 해시·단조성)
- 기존 데이터만 소비 → 마이그레이션·프롬프트 변경·AI 비용 0
- 스타일라이즈드 → 기대-실망 갭 구조적 완화

### 부정 / 리스크

- 절차 생성 메시의 심미성은 튜닝 노동에 비례 — "못생기면 역효과". 시각 검증 사이클 필요 (스냅샷 테스트 + 수동 확인)
- 최신 프로드 행 기준 입력 밀도가 낮음 (S/W/N + ratio 1개) → v1 아바타의 개인화 폭 제한. Tier 2(BodyRatios 저장)로 해소 경로 확보
- Three.js 도입 = 신규 대형 의존성 — avatar 모듈 밖 사용 금지 (P8 경계)

### 후속 항목

1. ~~`body_assessments` 유령 테이블 수리~~ → **본 작업 범위에 포함 확정** (2026-07-08): body-v2 단독 라우트 저장을 `body_analyses`로 정렬 (유령 배선 #13). 아바타가 읽는 데이터 경로의 상류이므로 함께 수리
2. `measurement_source` 마이그 gap-apply — Tier 2 마이그레이션 적용 시 함께 gap-apply (범위 밖 대기 항목이었으나 동일 테이블 작업이므로 동반)
3. 모바일 패리티 (expo-gl 또는 2D 폴백) — 웹 v1 검증 후 (범위 밖)
