# ADR-097: 분석 결과 시각적 오버레이 + 익명 공유 아키텍처

## 상태

`accepted`

## 날짜

## 0. 궁극의 형태 (P1)

### 이상적 최종 상태

모든 분석 모듈(7개)의 결과가 사용자 사진 위에 직관적 시각 오버레이로 표시되고,
공유 시 프라이버시를 보호하는 익명 모드가 제공되어,
"전문가 수준의 시각적 분석 결과를 무료로 즉시 제공하고, 부담 없이 공유할 수 있는" 앱.

## 1. 맥락 (Context)

### 1.1 문제 정의

이룸의 7개 분석 모듈은 기능적으로 완성(23,000+ 테스트 통과)되었으나,
결과 표시가 **텍스트+점수 중심**이어서 두 가지 문제가 있다:

1. **분석 신뢰도 체감 부족**: 사용자 시뮬레이션 결과 신뢰도 평균 4.2/10 (현재) → 시각화 시 7.5/10 (+79%)
2. **공유 바이럴 장벽**: 공유 의향 평균 3.6/10 → 익명 모드 시 6.2/10 (+72%)

### 1.2 경쟁사 벤치마크

| 경쟁사       | 시각화 수준                       | 이룸 대비                       |
| ------------ | --------------------------------- | ------------------------------- |
| Perfect Corp | 14항목 존별 히트맵 + 탭 전환      | 이룸보다 시각적 우위            |
| Exbody       | 33관절 스켈레톤 + 각도 수치       | 전문 장비 기반, 이룸은 스마트폰 |
| becon        | 두피 문제 영역 마커 + 레이더 차트 | 단일 모듈, 이룸은 8모듈 통합    |
| TroveSkin    | 레이더 차트 + 또래 백분위         | 시계열 강점                     |

### 1.3 기술 자산 현황 (이미 있는 것)

| 자산                   | 위치                     | 활용 계획                   |
| ---------------------- | ------------------------ | --------------------------- |
| face-api.js 68-point   | PhotoMetricOverlayV2.tsx | 피부/헤어/메이크업 오버레이 |
| MediaPipe Pose 33관절  | body-v2/pose-detector.ts | 체형/자세 오버레이          |
| 12존 히트맵 좌표       | zone-heatmap-data.ts     | 피부 히트맵 렌더링          |
| 7존 메트릭 (모공 포함) | ZoneMetricsV2            | 존별 점수→색상 매핑         |
| V1 SVG 존 경로 (8개)   | PhotoMetricOverlay.tsx   | 반투명 영역 채우기          |
| 5테마 공유 카드        | AnalysisShareCard.tsx    | 익명 모드 확장 기반         |

### 1.4 사용자 시뮬레이션 핵심 발견 (10명 페르소나)

1. **시각화의 역설**: 히트맵은 신뢰도를 올리지만, 결점 시각화로 공유 의향은 하락할 수 있음
2. **크리에이터 vs 일반 사용자**: 정반대 니즈 (실제 사진 vs 익명). A+B 동시만 양쪽 충족
3. **강점 하이라이트가 공유의 열쇠**: "결점 지도"가 아닌 "강점 지도"로 프레이밍 시 공유 폭발

## 2. 결정 (Decision)

### D1: 모듈별 시각화 기술 매핑

각 모듈의 특성에 맞는 최적 시각화 방식을 선택한다.
**"하나의 기술로 통일"이 아닌 "모듈 특성에 맞는 최적 기술"을 선택하는 것이 핵심.**

| 모듈               | 시각화 기법                 | 렌더링 기술 | 기반 라이브러리  | 이유                                                        |
| ------------------ | --------------------------- | ----------- | ---------------- | ----------------------------------------------------------- |
| **피부 (S-1)**     | 12존 히트맵 오버레이        | **SVG**     | face-api.js      | 기존 V1 경로+V2 랜드마크 조립. SVG가 반투명+상호작용에 최적 |
| **체형 (C-1)**     | 스켈레톤 + 비율 수치        | **Canvas**  | MediaPipe Pose   | 33개 관절점 연결 렌더링, Canvas가 라인 드로잉에 효율적      |
| **자세**           | 기준선 vs 편차 비교         | **Canvas**  | MediaPipe Pose   | 체형과 동일 인프라 공유, 정렬선 렌더링                      |
| **헤어 (H-1)**     | 얼굴형 윤곽선 + 스타일 카드 | **SVG**     | face-api.js      | 턱라인(0-16) 부드러운 곡선, SVG path 최적                   |
| **메이크업 (M-1)** | 페이스 존 컬러 가이드       | **SVG**     | face-api.js      | 피부 오버레이 패턴 재활용, 영역별 색상 채움                 |
| **구강 (OH-1)**    | 제네릭 치아 도식            | **SVG**     | 없음 (정적 도식) | 랜드마크 감지 불가, 사전 정의 치아 SVG 사용                 |
| **퍼스널컬러**     | 변경 없음                   | -           | -                | 컬러 팔레트가 본질, 오버레이 불필요                         |

**렌더링 기술 선택 근거:**

- **SVG**: 반투명 영역 채우기, 상호작용(호버/클릭), 해상도 독립적, CSS 애니메이션 용이 → 피부/헤어/메이크업/구강
- **Canvas**: 관절점 연결 라인 드로잉, 실시간 좌표 계산, 이미지 위 직접 렌더링 → 체형/자세

### D2: MediaPipe Pose 통합 전략

Legacy `@mediapipe/pose` CDN → **`@mediapipe/tasks-vision` PoseLandmarker로 마이그레이션하지 않는다.**

**현황 인지**: 프로젝트에 Legacy(`@mediapipe/pose` CDN)와 Tasks Vision(`@mediapipe/tasks-vision` npm)이 **공존** 중.

- Legacy: `body-v2/pose-detector.ts` (체형/자세 포즈 추정)
- Tasks Vision: `cie-2/mediapipe-loader.ts` (얼굴 FaceLandmarker)
- 두 라이브러리는 **용도가 다르므로** 공존 유지. 동일 페이지에서 동시 로드되지 않음 (각각 분석 모듈 전용).

| 대안                             | 선택 | 이유                                                          |
| -------------------------------- | ---- | ------------------------------------------------------------- |
| Tasks Vision PoseLandmarker 통합 | ❌   | 마이그레이션 공수 대비 정확도 차이 미미. 출시 지연 위험       |
| Legacy CDN 유지 (현재)           | ✅   | 이미 동작하는 코드 존재. "작동하는 코드를 고치지 마라" (P4)   |
| 향후 통합                        | 📋   | v1.1에서 Pose도 Tasks Vision으로 통합 검토 (CDN 지원 중단 시) |

### D3: 익명 공유 모드 아키텍처

**제네릭 SVG 일러스트 방식을 채택한다.** 모듈별로 적합한 익명화 방식이 다르다.

| 모듈           | 익명 모드 시각                                                  | 구현 방식                                         |
| -------------- | --------------------------------------------------------------- | ------------------------------------------------- |
| **피부**       | 제네릭 얼굴 일러스트 (얼굴형 3종: 타원/둥근/각진) + 12존 히트맵 | SVG 일러스트 위에 동일 히트맵 컬러매핑            |
| **체형**       | 성별 무관 기하학적 체형 실루엣 (S/W/N 3종) + 비율 수치          | 미니멀 기하학적 SVG (ADR-084 neutral 기본값 준수) |
| **자세**       | 체형과 동일 실루엣 + 정렬선                                     | 체형 실루엣 재활용                                |
| **헤어**       | 얼굴형 윤곽선만 (무배경) + 스타일 카드                          | 윤곽 SVG (face-api.js 데이터 기반 생성)           |
| **메이크업**   | 제네릭 얼굴 일러스트 + 영역별 색상 스와치                       | 피부와 동일 일러스트 재활용                       |
| **구강**       | 제네릭 치아 도식 (원래부터 도식이므로 전환 불필요)              | -                                                 |
| **퍼스널컬러** | 현재 공유 카드 유지 (이미 얼굴 없음)                            | -                                                 |

**성별 중립화 원칙 (ADR-084 준수):**

- 일러스트 분류 축은 **형태 기반** (얼굴형/체형), 성별 기반이 아님
- 체형 실루엣은 성별 특징 없는 기하학적 도형 (어깨:힙 비율만 변형)
- 피부톤 변형은 명도 기반 3단계 (밝음/중간/어두움), 인종/성별 무관

**공통 UI:**

- 결과 페이지 하단 공유 바에 "내 사진 포함 / 일러스트로 공유" 토글
- 기본값: **일러스트로 공유** (프라이버시 우선, opt-in 원칙)
- 크리에이터는 "내 사진 포함" 모드로 전환하여 실제 오버레이 캡처 가능

### D4: 강점 하이라이트 프레이밍 (ADR-080 확장)

**히트맵/오버레이의 기본 프레이밍을 "강점 중심"으로 한다.**

- **기본값**: 좋은 부분 = 골드/에메랄드로 강조, 약한 부분 = 회색으로 은은하게
- **토글**: "전체 보기" 모드에서 결점도 빨강으로 표시 (전문가/상세 분석용)
- **공유 시**: 강점 모드 고정 (결점 노출 방지)
- **색맹 대응 (ADR-048 WCAG 1.4.11 준수)**: 색상만으로 정보 전달하지 않음
  - 강점: 골드/에메랄드 + **체크 아이콘(✓)** + 실선 테두리
  - 약점: 회색 + **점선 테두리** (전체 보기 모드에서 빨강 + 경고 아이콘)
  - Triple Encoding: 색상 + 아이콘/패턴 + 텍스트 라벨 (ADR-077 ConcernCard와 동일 원칙)

근거: 사용자 시뮬레이션에서 인플루언서(하영) + 일반 사용자(민지) 모두 "강점 시각화 시 공유 의향 대폭 상승" 반응

### D5: 스캐닝 애니메이션

분석 로딩 화면에 시네마틱 스캐닝 효과를 추가한다.

| 모듈               | 애니메이션                                      | 구현                   |
| ------------------ | ----------------------------------------------- | ---------------------- |
| 피부/메이크업/헤어 | 스캔 라인이 얼굴을 위→아래로 훑으며 포인트 등장 | CSS animation + SVG    |
| 체형/자세          | 스캔 라인이 전신을 위→아래로 훑으며 관절점 등장 | CSS animation + Canvas |
| 구강               | 치아 도식이 영역별로 순차 하이라이트            | SVG animation          |

**가이드라인:**

- "시네마틱"이 아닌 **"분석 과정 시각화"**로 프레이밍 (이룸의 "따뜻한 전문가" 톤 유지)
- duration: AI 응답 시간에 연동 (실제 분석 시간과 동기화, 과장 연출 금지)
- easing: ease-out (자연스러운 감속)
- 반복: 1회 (루프 금지)
- **우선순위**: 핵심 기능(히트맵/스켈레톤/익명 공유) 이후 구현. P4(단순화) 관점에서 MVP에서 제외 가능.

근거: 시뮬레이션에서 3명(지은/소영/하영)이 "분석 과정 애니메이션 = 콘텐츠 소재"로 언급. 구현 공수 낮음 (CSS animation).

- 내부 구현(좌표 계산, 색상 매핑) 외부 접근 금지

### D7: Canvas 접근성 보장 (ADR-048 WCAG 4.1.2)

Canvas 모듈(체형/자세)은 스크린 리더가 내용을 인식할 수 없으므로 **접근성 대안을 필수 제공**한다.

- Canvas 옆에 **sr-only 텍스트 대안** 배치: "어깨 기울기 2.3도, 골반 정렬 1.5도 편차"
- Canvas 위에 **투명 HTML 레이어**를 덧씌워 각 관절점에 `aria-label` 부착
- `AnalysisOverlayBase.tsx`에서 공통 시각 토큰(색상, 투명도, corner radius)을 정의하여 SVG/Canvas 간 시각적 일관성 보장
- 키보드 Tab 네비게이션: 존/관절점 간 순차 이동 가능

### D8: 모바일 포팅 전략

**웹 우선, 모바일은 v1.1에서 포팅한다.**

| 이유          | 상세                                                             |
| ------------- | ---------------------------------------------------------------- |
| face-api.js   | Web API(Canvas, DOM) 의존 → React Native 직접 사용 불가          |
| Canvas 렌더링 | RN에서는 `react-native-canvas` 또는 `react-native-skia` 필요     |
| 출시 일정     | 웹+모바일 동시는 공수 2배, 5/1 출시 목표 위험                    |
| ADR-086 선례  | "Thin Client 이식" 패턴으로 웹 완성 후 모바일 포팅이 검증된 패턴 |

### D9: ADR-077 ConcernCard와 Layer 관계

히트맵 오버레이는 **Layer 0.5 (Visual Scan)**로 배치한다. 기존 Layer 구조를 확장:

```
Layer 0: Emotional Hook (한 문장 요약 + 점수) — 기존 유지
Layer 0.5: Visual Scan (히트맵/스켈레톤 오버레이) — ★ 신규
Layer 1: Concern Overview (ConcernCard 그리드) — ADR-077 유지
Layer 2: Deep Metrics (MetricBarGaugeList) — 기존 유지
Layer 3: Action (성분/루틴/제품 추천) — 기존 유지
Layer 4: Trust (근거 + AI 고지) — 기존 유지
```

- Layer 0.5는 사용자 사진 + 오버레이를 보여주는 시각적 앵커
- Layer 1(ConcernCard)은 히트맵을 텍스트/아이콘으로 보완하는 상세 그리드
- 두 Layer는 **보완 관계** (시각 → 텍스트), 대체 관계 아님

## 3. 대안 (Alternatives Considered)

### A1: 렌더링 기술 — 전체 Canvas 통일 (기각)

- **장점**: 단일 렌더링 파이프라인, 이미지 위 직접 그리기 용이
- **단점**: 반투명 영역 호버/클릭 상호작용 어려움, CSS 애니메이션 불가, 접근성(aria) 지원 한계
- **기각 사유**: 피부 히트맵은 존별 상호작용(탭 시 상세 정보)이 핵심 → SVG가 필수

### A2: 렌더링 기술 — 전체 SVG 통일 (기각)

- **장점**: 일관된 기술 스택, CSS 애니메이션, 접근성 양호
- **단점**: 33개 관절점 라인 드로잉에 SVG path가 비효율적, 이미지 좌표 변환 복잡
- **기각 사유**: 체형/자세 스켈레톤은 Canvas drawLine이 직관적이고 성능 우수

### A3: 익명 모드 — AI 기반 아바타 생성 (기각)

- **장점**: 개인화된 아바타, 시각적 매력
- **단점**: Gemini 호출 추가 비용, 생성 시간 2-5초, 일관성 없음, 불쾌한 골짜기(Uncanny Valley) 위험
- **기각 사유**: P4(단순화). 제네릭 SVG 일러스트가 비용 0, 즉시 렌더링, 일관된 품질

### A4: 익명 모드 — 사진 블러/실루엣 자동 추출 (기각)

- **장점**: 사용자 실제 체형/얼굴형 반영
- **단점**: 블러 강도 조절 어려움(너무 약하면 식별, 너무 강하면 의미 없음), 실루엣 추출은 별도 세그멘테이션 모델 필요
- **기각 사유**: 추가 ML 모델 로드 = 번들/로딩 증가. 제네릭 일러스트가 더 일관되고 프라이버시 보장 확실

### A5: 포즈 라이브러리 — Tasks Vision PoseLandmarker 마이그레이션 (보류)

- **장점**: 최신 API, Face+Pose 동일 패키지, 10-15% 성능 향상
- **단점**: 기존 Legacy CDN 코드 전면 리팩토링 필요, 테스트 재작성
- **보류 사유**: 출시 전 마이그레이션 = 리스크. 기존 코드가 정상 동작하므로 v1.1로 연기

### A6: 전체 기능 출시 후 업데이트 (기각)

- **장점**: 출시 일정 준수
- **단점**: 시뮬레이션 결과 신뢰도 4.2/10은 "앱이 진짜 분석한 건지 의문"이라는 치명적 피드백. 시각화 없이 출시하면 초기 사용자 이탈 위험
- **기각 사유**: 시각화는 "있으면 좋은 것"이 아니라 "분석 앱의 본질적 신뢰 장치"

## 4. 결과 (Consequences)

### 긍정적

- 분석 신뢰도 체감 +79% (시뮬레이션 4.2→7.5)
- 공유 의향 +72% (시뮬레이션 3.6→6.2)
- 기존 코드 자산(face-api.js, MediaPipe Pose, V1/V2 오버레이) 최대 활용
- 크리에이터 + 일반 사용자 동시 만족 (실제 사진 모드 + 익명 모드)
- 경쟁사(Perfect Corp, Exbody) 수준의 시각적 결과 달성
- 이룸 차별점(8모듈 통합) 시각적 증명

### 부정적

- 출시 일정 연장 (시각화 7모듈 + 익명 공유 구현 공수)
- CDN 모델 로드로 초기 방문 시 추가 다운로드 (face-api ~6MB, Pose ~9MB, 캐싱 후 0)
- SVG 일러스트 제작 필요 (얼굴 3종, 체형 3종, 치아 1종)
- 컴포넌트 수 증가 (overlay/ 폴더 ~10개 신규)

### 리스크 완화

| 리스크             | 완화 방안                                            |
| ------------------ | ---------------------------------------------------- |
| 출시 지연          | Phase 3 병렬화 (A/B/C 그룹), 피부 히트맵 MVP 우선    |
| CDN 로드 시간      | 분석 결과 페이지에서만 lazy load, 메인 번들 영향 0   |
| 일러스트 품질      | 미니멀 SVG (복잡한 일러스트 아닌 기하학적 실루엣)    |
| 강점 프레이밍 오해 | "결점을 숨기는 것이 아님" 안내 문구 + 전체 보기 토글 |

## 5. 구현 계획

### Phase 구조

```
Phase 1: 공통 인프라 (직렬)
  → AnalysisOverlayBase, ScanningAnimation, StrengthHighlightToggle
  → 예상: 1세션

  → 그룹 B (MediaPipe Pose): 체형 스켈레톤 + 자세 정렬선
  → 그룹 C (정적 SVG): 구강 치아 도식
  → 예상: 2세션 (병렬 시 1.5세션)

Phase 3: 익명 공유 모드 (★ 병렬 가능)
  → AnonymousFaceTemplate + AnonymousBodyTemplate
  → 공유 바 토글 UI
  → 예상: 1세션

Phase 4: 통합 + 테스트 (직렬)
  → 모듈별 결과 페이지에 오버레이 통합
  → typecheck + lint + test
  → 예상: 1세션
```

- **트랙**: Full (71-100점) — 20+ 파일, 새 패턴 도입, 7모듈 걸침
- **병렬 에이전트**: Phase 2에서 그룹 A/B/C 동시 실행
