# UX 디자인 원칙

> 새 기능 설계 시 참조하는 철학 레퍼런스
> 체크리스트가 아닌 **내면화 문서** — 읽고 방향을 잡는 용도
> 자동 점검 항목은 [ux-pr-checklist.md](./ux-pr-checklist.md) 참조

---

## 1. 제품 판단의 6대 테스트

### Jobs 6 Tests

> 잡스의 제품 철학을 6가지 독립 테스트로 구조화

#### T1. 삭제 테스트 (1,000 No's)

> "Focus is about saying no to the hundred other good ideas."

| 질문                                  | 판단 기준                        |
| ------------------------------------- | -------------------------------- |
| 이 기능을 **완전히 삭제하면?**        | 핵심 가치가 훼손되지 않으면 삭제 |
| **6살짜리**도 사용법을 알 수 있나?    | 설명이 필요하면 디자인이 틀림    |
| 이 화면에서 **가장 중요한 것 1개**는? | 즉답 못 하면 초점이 없음         |

#### T2. 취향 테스트 (Taste)

> "The only problem with Microsoft is they just have no taste."

| 질문                                        | 판단 기준                             |
| ------------------------------------------- | ------------------------------------- |
| 이게 우리가 만들 수 있는 **최선**인가?      | "충분히 좋다"는 답이면 부족           |
| 이걸 사용하면서 **자부심**을 느끼는가?      | 제작자가 자랑하고 싶지 않으면 재설계  |
| 경쟁 앱이 아닌 **최고의 앱**과 비교했을 때? | 같은 카테고리가 아닌 최고 수준과 비교 |

#### T3. 1초 테스트 (Glance Test)

> "Design is not just what it looks like. Design is how it works."

| 질문                                          | 판단 기준                                     |
| --------------------------------------------- | --------------------------------------------- |
| **1초** 만에 이 화면의 목적이 보이는가?       | 1초 후 화면을 가렸을 때 핵심을 말할 수 있어야 |
| 시선이 자연스럽게 **핵심→행동**으로 흐르는가? | 눈이 헤매면 시각적 위계 실패                  |
| 장식과 핵심을 **즉시 구분**할 수 있는가?      | 구분 못 하면 장식이 핵심을 가린 것            |

#### T4. End-to-End 테스트

> "We do these things not because we are control freaks... because we want to make great products."

| 질문                                          | 판단 기준                     |
| --------------------------------------------- | ----------------------------- |
| 이전 화면에서 여기까지 **전환이 매끄러운가?** | 맥락 단절이 있으면 실패       |
| 이 화면 후 **다음 단계가 자연스러운가?**      | "이제 뭐 하지?"가 나오면 실패 |
| 첫 설치부터 지금까지 **하나의 이야기**인가?   | 기능 모음이 아닌 경험의 흐름  |

#### T5. Dent 테스트

> "We're here to put a dent in the universe."

| 질문                                                 | 판단 기준                             |
| ---------------------------------------------------- | ------------------------------------- |
| 이 기능이 사용자 삶에 **의미 있는 변화**를 만드는가? | 편의가 아닌 변화                      |
| 이룸 없이는 **불가능했을 경험**인가?                 | 다른 앱으로 대체 가능하면 차별성 부족 |
| 사용자가 **다른 사람에게 말하고 싶은** 경험인가?     | "이 앱이 나를 이렇게 알더라" 반응     |

#### T6. Magic 테스트

> "People don't know what they want until you show it to them."

| 질문                                      | 판단 기준                                  |
| ----------------------------------------- | ------------------------------------------ |
| 사용자가 **"와!"** 하는 순간이 있는가?    | 기대 이상의 인사이트, 예상 못 한 연결      |
| **요청하기 전에** 필요한 것을 보여주는가? | 검색이 아닌 발견의 경험                    |
| 기술이 **보이지 않게** 작동하는가?        | AI가 분석했다는 느낌이 아닌, 자연스러운 앎 |

> **주의**: T6은 포커스 그룹이 아닌 **비전과 취향**으로 판단한다.
> 잡스는 "사람들은 보여주기 전까지 원하는 걸 모른다"고 했다.
> 사용자가 요청한 것 = 좋은 기능, 사용자가 상상 못 한 것 = 위대한 기능.

### 잡스-이룸 교차점

> 잡스: "Technology × Liberal Arts" → Apple
> 이룸: "AI × Self-Understanding" → 통합 웰니스

| 잡스 원칙            | 이룸 적용                           | 검증 질문                                          |
| -------------------- | ----------------------------------- | -------------------------------------------------- |
| 기술은 보이지 않아야 | AI 분석은 "자연스러운 깨달음"이어야 | "AI가 분석했다"가 아닌 "나를 알게 됐다" 느낌인가?  |
| 인문학적 감성        | 자기 이해는 데이터가 아닌 이야기    | 숫자 나열이 아닌 "당신의 이야기"로 전달하는가?     |
| 통합된 경험          | 9개 모듈이 하나의 자기 이해         | 모듈 간 연결이 "도구 모음"이 아닌 "하나의 나"인가? |
| 단순함의 정교함      | 복잡한 분석을 단순한 인사이트로     | 전문 지식 없이 "아, 그렇구나"가 나오는가?          |

### Rams Test

> "좋은 디자인은 가능한 한 최소한의 디자인이다"

| 질문                                     | 판단 기준              |
| ---------------------------------------- | ---------------------- |
| 이 요소가 제품을 **더 유용하게** 만드나? | 장식이면 제거          |
| **10년 후**에도 촌스럽지 않을까?         | 트렌디한 효과면 단순화 |
| 모든 디테일에 **의도**가 있나?           | 답할 수 없으면 재설계  |

### Norman Test

> "The Design of Everyday Things" 6대 원칙

| 원칙                 | 질문                                         |
| -------------------- | -------------------------------------------- |
| **Affordance**       | 각 요소가 "나를 탭해"를 스스로 말하는가?     |
| **Mapping**          | 동작과 결과의 관계가 자연스러운가?           |
| **Feedback**         | 모든 행동에 즉각 피드백이 있나? (0.1초 이내) |
| **Conceptual Model** | 사용자의 멘탈 모델과 앱 구조가 일치하나?     |
| **Constraints**      | 잘못된 입력을 사전에 차단하는가?             |
| **Discoverability**  | 모든 기능이 발견 가능한가?                   |

### Ive Test

> "다른 방식으로 만들 수 없었을 것 같은" 디자인

- 이 디자인이 **유일하게 올바른 형태**처럼 느껴지는가?
- 불필요한 레이어나 추상화 없이 **정직한가?**
- 보이지 않는 곳(에러, 극단 데이터)에도 **동일한 수준의 배려**가 있는가?

## 2. 이룸 3-Layer 가치

> 모든 기능은 3-Layer 중 하나 이상에 기여해야 한다
> 근거: [yiroom-philosophy.md](../principles/yiroom-philosophy.md) Section 3

| Layer               | 질문                           | 예시                                   |
| ------------------- | ------------------------------ | -------------------------------------- |
| **L1: 즉각 해결**   | 구체적 궁금증을 해소하는가?    | "내 퍼스널컬러가 뭐지?"                |
| **L2: 판단력 성장** | **"왜"**를 설명하는가?         | "봄 웜톤**이라서** 코랄 립이 어울려요" |
| **L3: 자기 이해**   | 통합적 자기 이해에 기여하는가? | "피부+컬러+체형이 이렇게 연결돼요"     |

**핵심 질문**:

- L1만 제공하는 기능이 아닌가? (L2 이상 필수)
- 사용자의 **자립**을 돕는가, **의존**을 만드는가?

### Fogg 행동 모델과의 연결 (B=MAP)

> 근거: [yiroom-philosophy.md](../principles/yiroom-philosophy.md) Section 3.2

```
행동(B) = 동기(M) × 능력(A) × 프롬프트(P)
```

- **동기**: 이미 존재 (앱을 설치했으므로)
- **능력**: 이룸의 핵심 병목 — "왜 이게 나한테 맞는지" 모름
- **프롬프트**: 적절한 타이밍에 적절한 인사이트

**이룸의 역할 = 능력(Ability) 향상**. 교육이 아닌 경험으로:

- "봄 웜톤 이론을 가르친다" (X) → "코랄 립 추천 시 '웜톤이라서'를 보여준다" (O)
- 이것이 E1(인과 연결)과 E2(L2 가치)의 근본 이유다.

## 3. 감정 설계 가이드

> 이룸의 감정 = 호기심 → 발견 → 성장 → 자신감
> 근거: [emotional-design-guide.md](../principles/emotional-design-guide.md)

### 허용 감정

| 감정        | 트리거                               | UX 라이팅 톤        |
| ----------- | ------------------------------------ | ------------------- |
| 호기심      | 미지의 자기 정보, 교차 인사이트 예고 | 질문형, 가벼운 초대 |
| 발견의 기쁨 | 분석 결과 최초 확인                  | 긍정적 놀람         |
| 성장 실감   | 시간 경과 후 변화, 내재화 진행       | 진심어린 축하       |
| 자신감      | 자립적 판단 성공                     | 진정한 인정         |

### 금지 감정 → 대체 패턴

| 감정   | 금지 패턴                 | 대체 패턴                                  |
| ------ | ------------------------- | ------------------------------------------ |
| 불안   | "지금 관리하지 않으면..." | "변화가 있네요. 이런 관리를 추가해볼까요?" |
| 열등감 | "평균보다 낮아요"         | "현재 상태에서 이런 개선이 가능해요"       |
| 강박   | "오늘의 분석을 놓쳤어요!" | "다시 와주셔서 반가워요"                   |
| 비교   | "같은 나이대 평균"        | 오직 자기 자신의 과거 데이터와 비교        |

### Delight 설계 (Magic Moment)

> 잡스 T6: "사람들은 보여주기 전까지 원하는 걸 모른다"
> 금지 감정을 피하는 것만으로는 부족하다. **"와!" 순간**을 의도적으로 설계한다.

| 순간          | 트리거                                | 설계 방향                                            |
| ------------- | ------------------------------------- | ---------------------------------------------------- |
| **첫 발견**   | 첫 분석 결과 확인                     | 예상 못 한 정확한 인사이트로 "이 앱 나를 아네?" 반응 |
| **교차 연결** | 2번째+ 분석 후 모듈 간 연결           | "피부랑 컬러가 이렇게 연결되다니!" — Aha Moment      |
| **성장 인지** | ConnectionAwareness internalized 달성 | "나도 모르게 판단력이 생겼네" — 조용한 자신감        |
| **숨은 배려** | 에러/극단 상황에서도 정중한 대응      | "이 앱은 나를 존중하는구나" — 신뢰의 순간            |

**핵심**: Delight는 화려한 애니메이션이 아니라 **"이 앱이 나를 이해한다"**는 느낌이다.

### 신체 이미지 보호 4대 원칙

1. 결핍 프레이밍 금지 ("부족하다", "안 좋다")
2. 타인 비교 금지 ("평균", "이상적", "표준" 체형)
3. 부정적 신체 언어 금지 ("문제 부위", "약점", "결점")
4. 모든 분석 결과 후 **실행 가능한 긍정 액션** 제시

### 프라이버시 신뢰 원칙

> 이룸은 얼굴·신체·구강 이미지를 다룬다. 사용자 신뢰 없이는 첫 분석도 시작되지 않는다.

1. 이미지 수집 시점에 **데이터 처리 방식을 명시** ("분석 후 즉시 삭제", "서버에 저장하지 않아요")
2. **최소 수집 원칙** — 분석에 필요한 정보만 요청, 불필요한 권한 요구 금지
3. 기술적 보안이 아닌 **사용자가 느끼는 안전감** 설계 — "내 사진이 안전하다"는 확신
4. 프라이버시 안내는 숨김 메뉴가 아닌 **업로드 화면에 직접 노출**

## 4. 가치 실현 가이드 (능동적 체크)

> 위의 원칙은 "방향"이고, 이 섹션은 "코드에서 확인 가능한 가치 실현"이다.
> 금지 패턴(D1~D4)만으로는 부족하다. **"해야 할 것"**을 명시한다.
> 각 항목은 [ux-pr-checklist.md](./ux-pr-checklist.md)의 체크리스트 항목과 1:1 매핑된다.

### 4.1 허용 감정 흐름 유도 → D7

> 금지 감정을 피하는 것만으로는 "무색무취한 앱"이 된다. 의도적으로 긍정 감정을 설계한다.

| 감정 단계  | 트리거                 | UX 실현 방법                                          | 코드 확인 포인트                         |
| ---------- | ---------------------- | ----------------------------------------------------- | ---------------------------------------- |
| **호기심** | 아직 모르는 자기 정보  | 질문형 텍스트 ("~해볼까요?", "~궁금하지 않나요?")     | 홈/분석 입력 화면의 CTA 텍스트           |
| **발견**   | 분석 결과 최초 확인    | 인사이트 텍스트 ("~이라서 ~해요", "~와 ~이 연결돼요") | 결과 페이지의 description/insight 텍스트 |
| **성장**   | 시간 경과 후 변화 인지 | 변화 축하 ("이전보다 ~해졌어요", 내재화 진행도)       | ConnectionAwareness 상태 전이 UI         |
| **자신감** | 자립적 판단 성공       | 자립 인정 ("이제 스스로 판단할 수 있어요")            | ExplanationDepth `none` 상태의 UI        |

**핵심**: 4단계 중 **최소 1개 이상**이 모든 주요 화면에 존재해야 한다.
정보만 나열하고 감정 유도 요소가 전혀 없는 화면은 "무색무취한 도구"에 불과하다.

### 4.2 긍정 액션 필수 제시 → D8

> 신체 이미지 보호 4대 원칙의 실행 가능한 형태

| 상황           | 금지 패턴                 | 올바른 패턴                                  |
| -------------- | ------------------------- | -------------------------------------------- |
| 피부 점수 낮음 | "피부 상태가 좋지 않아요" | "수분 크림을 추가하면 더 촉촉해질 수 있어요" |
| 체형 분석 후   | "하체가 약해요"           | "이 스타일링이 비율을 살려줄 수 있어요"      |
| 구강 점수      | "치아 상태 위험"          | "이 관리법으로 더 건강하게 유지해보세요"     |
| 운동 부족      | "운동이 부족해요"         | "가벼운 스트레칭부터 시작해볼까요?"          |

**원칙**: 모든 분석 결과 뒤에는 **"이렇게 해보세요"**가 따라와야 한다.
점수·수치만 보여주고 끝나는 화면은 불안을 유발할 수 있다.

### 4.3 L3 통합 자기 이해 → E5

> 이룸의 궁극적 가치: "나는 하나의 존재이고, 모든 분석은 하나로 연결된다"

| Layer                   | 화면 예시                              | 체크 기준                          |
| ----------------------- | -------------------------------------- | ---------------------------------- |
| **L1** (즉각 해결)      | "봄 웜톤이에요"                        | 단일 모듈 결과 → 이것만으로는 부족 |
| **L2** (판단력 성장)    | "봄 웜톤**이라서** 코랄 립이 어울려요" | 인과 설명 → E1, E2로 커버          |
| **L3** (통합 자기 이해) | "피부+컬러+체형이 **이렇게 연결돼요**" | 2개+ 모듈 교차 → **이것이 E5**     |

**실현 위치**: 홈(인사이트 카드), 대시보드(종합 리포트), 분석 완료 후(교차 인사이트)
**위험 신호**: 9개 모듈이 각각 독립적으로만 표시되고 연결이 전혀 없으면, "도구 모음"에 불과하다.

### 4.4 AI 비가시성 → E6

> 잡스-이룸 교차점: "기술은 보이지 않아야 한다"

| 금지 표현          | 대체 표현               | 이유                        |
| ------------------ | ----------------------- | --------------------------- |
| "AI가 분석한 결과" | "당신의 피부 분석 결과" | 주어는 AI가 아닌 **사용자** |
| "AI 추천 제품"     | "맞춤 추천 제품"        | 기술이 아닌 **가치** 중심   |
| "인공지능 코치"    | "나만의 웰니스 가이드"  | 기술 용어가 아닌 **역할**   |
| "Gemini 분석"      | (노출 금지)             | 내부 기술 스택 노출 금지    |

**핵심**: 사용자는 "AI가 분석했다"가 아닌 **"나를 알게 됐다"**를 느껴야 한다.
기술은 마법처럼 보이지 않게 작동해야 한다 (Jobs T6: Magic Test).

### 4.5 이야기형 전달 → E7

> "숫자 나열이 아닌 당신의 이야기"

| 숫자 나열 (L1)     | 이야기 전달 (L2+)                                                                  |
| ------------------ | ---------------------------------------------------------------------------------- |
| "피부 수분: 65점"  | "당신의 피부는 수분이 조금 부족한 편이에요. 환절기에 더 신경 쓰면 좋아요"          |
| "봄 웜톤 / 라이트" | "따뜻한 봄 빛이 잘 어울리는 타입이에요. 코랄과 피치 계열이 얼굴을 환하게 밝혀줘요" |
| "체형: 역삼각형"   | "어깨 라인이 돋보이는 체형이에요. V넥이나 보트넥으로 균형을 살릴 수 있어요"        |
| "BMR: 1,520kcal"   | "하루에 기본적으로 1,520kcal이 필요해요. 가벼운 활동까지 포함하면 더 늘어나요"     |

**원칙**: 모든 수치에는 **맥락과 의미**가 붙어야 한다.
숫자만 보여주면 사용자는 "그래서 뭐?"라고 느낀다.

### 4.6 핵심 원칙 → 체크리스트 전체 매핑

| 이룸 핵심 원칙       | 금지 체크 (하지 말 것)                   | 실현 체크 (해야 할 것)                                |
| -------------------- | ---------------------------------------- | ----------------------------------------------------- |
| **감정 설계**        | D1 불안, D2 열등감, D3 강박, D4 신체부정 | D5 Delight, **D7 허용감정**, **D8 긍정액션**          |
| **3-Layer 가치**     | —                                        | E1 인과연결(L2), E2 왜 설명(L2), **E5 통합 이해(L3)** |
| **자립 지향 (SDT)**  | E3 의존 유도 금지, **D9 자율성 침해**    | E4 시간축 적응 (내재화 추적), **D9 선택지 제안**      |
| **잡스-이룸 교차점** | **E6 AI 비가시성**                       | **E7 이야기형 전달**, D5 Delight                      |
| **신체 이미지 보호** | D4 부정 언어 금지                        | **D8 긍정 액션 필수**                                 |
| **프라이버시 신뢰**  | D6 안내 미존재 금지                      | D6 프라이버시 안내 존재                               |
| **윤리 (AI 한계)**   | **D10 비의료 고지 누락**                 | **D10 면책 안내 존재**                                |
| **비교 원칙**        | D2 타인 비교 금지                        | (과거의 나와만 비교 — D2 탐지에 포함)                 |

> **v5.0 균형**: 금지 8항목 + 실현 8항목 = **16항목이 이룸 철학을 직접 체크**

## 5. 자기결정이론 (SDT) 적용 → D9

> 근거: [yiroom-philosophy.md](../principles/yiroom-philosophy.md) Section 3.1
> "사용자가 스스로 선택했다는 느낌"이 참여와 내재화의 핵심이다.

### 3대 심리 욕구

| 욕구                     | 이룸 적용                           | UX 실현                                        |
| ------------------------ | ----------------------------------- | ---------------------------------------------- |
| **자율성 (Autonomy)**    | 추천은 "제안"이지 "지시"가 아니다   | "~해보면 좋아요", "~은 어때요?" (D9)           |
| **유능감 (Competence)**  | 사용자 판단력이 성장한다는 실감     | ConnectionAwareness 내재화 추적, 자립 마일스톤 |
| **관계성 (Relatedness)** | 앱이 "나를 이해하는 친구" 같은 느낌 | 해요체, 발견의 기쁨 공유 톤                    |

### 자율성 설계 원칙

| 금지 (지시형)          | 허용 (제안형)                |
| ---------------------- | ---------------------------- |
| "이 제품을 사용하세요" | "이 제품은 어때요?"          |
| "반드시 ~해야 해요"    | "~해보면 좋아요"             |
| "이걸로 바꾸세요"      | "이런 대안도 있어요"         |
| "매일 분석하세요"      | "여유 있을 때 해보면 좋아요" |

**핵심**: 사용자가 "시킨 대로 했다"가 아닌 **"내가 선택했다"**를 느껴야 한다.

### 유능감 설계 원칙

> 사용자가 "나도 할 수 있다"를 느끼게 한다. D7(허용감정 — 성장/자신감)으로 간접 점검.

| 금지 (AI 주도)           | 허용 (사용자 주도)                     | 코드 확인 포인트                                |
| ------------------------ | -------------------------------------- | ----------------------------------------------- |
| "AI가 추천합니다"        | "지난번에 직접 고른 컬러가 정확했어요" | 자립 인정 텍스트 존재 여부 (E4 상태)            |
| "최적의 제품을 찾았어요" | "이 조합, 잘 고르셨어요"               | ConnectionAwareness `internalized` 상태 축하 UI |
| 항상 동일한 설명 깊이    | 내재화 진행에 따라 설명 축소           | ExplanationDepth 분기 (E4)                      |

### 관계성 설계 원칙

> 앱이 "나를 이해하는 친구" 같은 느낌. A2(해요체)로 간접 점검.

| 금지 (기계적)            | 허용 (공감적)                         | 코드 확인 포인트         |
| ------------------------ | ------------------------------------- | ------------------------ |
| "분석 결과를 표시합니다" | "당신의 피부에 대해 알게 됐어요"      | 해요체 + 2인칭 표현 (A2) |
| "오류가 발생했습니다"    | "잠시 문제가 생겼어요. 다시 해볼게요" | 에러 메시지 톤 (C2)      |
| "데이터 없음"            | "아직 분석 전이에요. 시작해볼까요?"   | 빈 상태 톤 (C3)          |

## 6. 윤리 원칙 → D10

> 근거: [yiroom-philosophy.md](../principles/yiroom-philosophy.md) Section 9

### AI 한계 고지

이룸은 AI 기반 분석을 제공하지만, **의료 진단을 대체하지 않는다**.

| 분석 유형 | 필수 고지                                                            |
| --------- | -------------------------------------------------------------------- |
| 피부 분석 | "이 분석은 참고용이며, 피부 질환 진단은 전문의와 상담하세요"         |
| 구강 건강 | "구강 건강 참고 정보이며, 치과 전문의의 진단을 대체하지 않아요"      |
| 영양 분석 | "일반적인 영양 정보이며, 의료 목적의 식이요법은 전문가와 상의하세요" |

### 어필리에이트 투명성

> 근거: 철학 Section 8.2

| 금지 패턴                | 올바른 패턴                                       |
| ------------------------ | ------------------------------------------------- |
| "인기 상품이에요" (단독) | "봄 웜톤이라서 이 컬러가 잘 어울려요" (인과 포함) |
| 매출 최적화 순위 노출    | 사용자 특성 기반 매칭 순위                        |
| 제휴 여부 미표시         | "제휴 링크" 표시 (투명성)                         |

**원칙**: 추천에는 **항상 "왜 당신에게 맞는지"** 설명이 포함되어야 한다 (E1 확장).

## 7. Aha Moment & 비즈니스 정렬

> 근거: [yiroom-philosophy.md](../principles/yiroom-philosophy.md) Section 5, 8

### Aha Moment 설계

> "2번째 분석 완료 시 교차 인사이트" = 이룸의 핵심 Aha Moment

| 단계           | 사용자 경험                                    | 설계 목표                    |
| -------------- | ---------------------------------------------- | ---------------------------- |
| 1번째 분석     | "내 퍼스널컬러가 봄 웜톤이구나" (L1)           | 즉각적 가치 → 호기심 유발    |
| **2번째 분석** | "피부 타입이랑 컬러가 이렇게 연결되다니!" (L3) | **Aha Moment → 리텐션 핵심** |
| 3번째+         | "나는 하나의 존재, 모든 게 연결돼"             | 통합적 자기 이해 → 자립      |

**비즈니스 정렬**: Aha Moment까지의 경로가 짧을수록 리텐션이 높다.

- G3 체크리스트 항목이 이를 점검한다.
- 첫 분석 완료 후 **"다음 분석 해볼까요?"** CTA가 Aha까지의 최단 경로.

### 성장 가시화

사용자가 **"이 앱 덕에 나를 더 잘 알게 됐어"**라고 느끼면 자연스러운 리텐션이 발생한다.
DAU를 위해 강박을 유발하는 것이 아니라, **가치를 통해 재방문**하게 한다.

| 반리텐션 패턴 (금지)     | 가치 리텐션 패턴 (권장)        |
| ------------------------ | ------------------------------ |
| "오늘 분석 안 하셨어요!" | "새로운 인사이트가 도착했어요" |
| "연속 기록이 끊어졌어요" | "지난달보다 판단력이 늘었어요" |
| "친구들은 다 했어요"     | "당신만의 새 발견이 있어요"    |

## 8. 반지표 (Anti-Metrics)

> 근거: [yiroom-philosophy.md](../principles/yiroom-philosophy.md) Section 10
> "측정하는 것이 행동을 결정한다" — 잘못된 지표는 잘못된 설계를 유도한다.

### 추적하지 않는 지표

| 반지표         | 왜 추적하지 않는가       | 대신 추적하는 것                          |
| -------------- | ------------------------ | ----------------------------------------- |
| DAU 체류 시간  | 강박적 사용 유도 위험    | **가치 도달률** (Aha Moment까지의 전환)   |
| 알림 클릭률    | 빈번한 알림 → 강박       | **인사이트 가치 알림만** (가치 있을 때만) |
| 스트릭 길이    | 연속 기록 압박 → D3 위반 | **내재화 진행도** (ConnectionAwareness)   |
| 타인 비교 순위 | 열등감 유발 → D2 위반    | **과거의 나와 비교** (성장 실감)          |
| 분석 횟수/일   | 과분석 강박              | **인사이트 품질** (교차 연결 수)          |

### 건강한 지표

| 지표                  | 측정 방법                                | 목표                |
| --------------------- | ---------------------------------------- | ------------------- |
| **Aha Moment 도달률** | 2번째 분석 후 교차 인사이트 노출 비율    | > 70%               |
| **내재화 전환률**     | exposed → internalized 전이 비율         | 자연스러운 성장     |
| **자립 달성률**       | ExplanationDepth `none` 도달 사용자 비율 | 점진적 증가         |
| **재방문 이유**       | "새 인사이트" vs "습관/강박"             | 인사이트 기반 > 80% |

**원칙**: 지표가 **사용자의 성장**을 측정하면 올바른 설계가 따라온다.
지표가 **앱의 점유**를 측정하면 강박과 불안을 유발하는 설계가 따라온다.

## 9. Nielsen 10 Heuristics (구조 점검용)

> 변경된 화면에 대해서만 빠르게 훑는다. 심각도 3~4만 즉시 수정.

| #   | 휴리스틱            | 이룸 적용 질문                         |
| --- | ------------------- | -------------------------------------- |
| H1  | 시스템 상태 가시성  | 분석 진행 중 프로그레스 바 있는가?     |
| H2  | 시스템-실세계 일치  | 시스템 용어가 UI에 노출되지 않는가?    |
| H3  | 사용자 통제와 자유  | 실수로 탭했을 때 돌아갈 수 있는가?     |
| H4  | 일관성과 표준       | 같은 기능이 곳곳에서 다른 아이콘/이름? |
| H5  | 에러 예방           | 잘못된 입력을 사전에 차단하는가?       |
| H6  | 인식 > 기억         | 이전 결과를 기억 없이 비교 가능한가?   |
| H7  | 유연성과 효율       | 숙련 사용자가 빠르게 원하는 곳으로?    |
| H8  | 미니멀 디자인       | 각 위젯이 하나의 목적에 집중?          |
| H9  | 에러 인식/진단/복구 | 에러 메시지가 해결책을 제시?           |
| H10 | 도움말과 문서       | 복잡한 기능에 툴팁/안내 있는가?        |

**심각도**: 0(위반 아님) ~ 4(사용 불가). 3~4만 즉시 수정, 1~2는 다음 Cycle.
