## 0. 궁극의 형태 (P1)

### 이상적 최종 상태

```
"모든 리스크가 사전에 식별되고,
각 리스크에 대한 완화 계획이 수립되어,
위기 발생 시 즉각 대응 가능한 상태"
```

| 영역 | 궁극의 형태 |
|------|------------|
| **식별** | 모든 잠재 리스크가 매트릭스에 등록 |
| **평가** | 영향도 × 발생확률로 정량화 |
| **완화** | 각 리스크별 대응 계획 문서화 |
| **모니터링** | 분기별 리뷰로 리스크 상태 업데이트 |

### 현재 달성도: **65%**

| 항목 | 목표 | 현재 | 갭 |
|------|------|------|-----|
| 리스크 식별 | 전체 영역 커버 | 3개 영역 ✅ | - |
| 정량화 평가 | 모든 리스크 | 95% | -5% |
| 완화 계획 | 상위 리스크 100% | 80% | 개선 필요 |
| 모니터링 사이클 | 자동화 | 수동 | 향후 |

### 의도적 제외

| 항목 | 이유 | 재검토 시점 |
|------|------|------------|
| 지정학적 리스크 | 국내 시장 집중 | 글로벌 확장 시 |
| M&A 리스크 | Pre-seed 단계 | Series B |
| 공급망 리스크 | 소프트웨어 제품 | 하드웨어 연동 시 |

## 1. 목적

```
"리스크를 회피하는 것이 아니라 관리하여,
불확실성 속에서도 지속 가능한 성장을 추구"
```

### 1.1 왜 필요한가? (P0 적용)

| 질문 | 답변 |
|------|------|
| 1인 스타트업에 리스크 관리가 필요한가? | **필수** - 단일 실패점(SPOF) 많음 |
| 삭제하면 어떻게 되는가? | 위기 발생 시 대응 불가, 투자자 신뢰 하락 |
| 더 단순한 방법은? | 상위 리스크만 집중 관리 (80/20 원칙) |

### 1.2 Solo Founder 특수 상황

```
일반 스타트업 vs Solo Founder:

일반: 역할 분담 → 리스크 분산
Solo: 모든 역할 집중 → 리스크 집중

핵심 차이:
- 키맨 리스크 = 100% (분산 불가)
- 의사결정 병목 = 창업자 1명
- 번아웃 = 사업 중단과 동일
```

## 2. 리스크 분류 체계

### 2.1 영향도 × 발생확률 매트릭스

```
                 발생확률
         1      2      3      4      5
        (희박) (낮음) (중간) (높음) (거의확실)
      ┌──────┬──────┬──────┬──────┬──────┐
    5 │  5   │  10  │  15  │  20  │  25  │ 치명적
영    │ 저   │ 중   │ 고   │ 심각 │ 심각 │
향  4 │  4   │  8   │  12  │  16  │  20  │ 높음
도    │ 저   │ 중   │ 중   │ 고   │ 심각 │
    3 │  3   │  6   │  9   │  12  │  15  │ 중간
      │ 저   │ 저   │ 중   │ 중   │ 고   │
    2 │  2   │  4   │  6   │  8   │  10  │ 낮음
      │ 저   │ 저   │ 저   │ 중   │ 중   │
    1 │  1   │  2   │  3   │  4   │  5   │ 경미
      │ 저   │ 저   │ 저   │ 저   │ 저   │
      └──────┴──────┴──────┴──────┴──────┘
```

### 2.2 리스크 등급 정의

| 등급 | 점수 | 색상 | 대응 수준 |
|------|------|------|----------|
| **심각** | 16-25 | Red | 즉시 완화 계획 실행, 주간 모니터링 |
| **고** | 12-15 | Orange | 완화 계획 수립 필수, 월간 모니터링 |
| **중** | 6-11 | Yellow | 완화 계획 검토, 분기 모니터링 |
| **저** | 1-5 | Green | 수용 가능, 연간 리뷰 |

### 2.3 영향도 척도

| 점수 | 수준 | 정의 |
|------|------|------|
| 5 | 치명적 | 사업 지속 불가, 회복 불가능 |
| 4 | 높음 | 심각한 재정 손실, 장기 회복 필요 |
| 3 | 중간 | 상당한 영향, 수개월 내 회복 가능 |
| 2 | 낮음 | 제한적 영향, 수주 내 회복 가능 |
| 1 | 경미 | 미미한 영향, 즉시 회복 가능 |

### 2.4 발생확률 척도

| 점수 | 수준 | 정의 |
|------|------|------|
| 5 | 거의 확실 | 1년 내 발생 확실 (>80%) |
| 4 | 높음 | 1년 내 발생 가능성 높음 (50-80%) |
| 3 | 중간 | 1-2년 내 발생 가능 (20-50%) |
| 2 | 낮음 | 2-3년 내 발생 가능 (5-20%) |
| 1 | 희박 | 발생 가능성 매우 낮음 (<5%) |

## 6. 완화 전략

### 6.1 전략 유형

| 전략 | 설명 | 적용 상황 |
|------|------|----------|
| **회피** | 리스크 발생 원인 제거 | 리스크 가치 < 비용 |
| **완화** | 영향도 또는 확률 감소 | 가장 일반적 |
| **전가** | 제3자에게 리스크 이전 | 보험, 외주 |
| **수용** | 리스크 감수 | 영향 작거나 비용 과다 |

### 6.2 기술 리스크 완화

#### T-1: AI API 의존성 완화

```
전략: 완화 + 일부 회피

단기 (현재):
✅ Mock Fallback 시스템 구축 (완료)
✅ 3단계 폴백: 재시도 → Mock → 우아한 실패
✅ Feature Flag로 AI 기능 토글 가능

중기 (6개월):
□ 멀티 AI 지원 (Claude, GPT 백업)
□ 오프라인 경량 모델 연구
□ 캐싱 레이어 강화

장기 (1년+):
□ 자체 모델 파인튜닝 검토
□ AI 비용 최적화 (배치 처리)

비용: 개발 2주 + API 비용 20% 증가 (백업용)
책임자: CTO 기능 (창업자)
```

#### T-3: 보안 취약점 완화

현재 조치:
✅ Supabase RLS 정책 적용
✅ Clerk 인증 통합
✅ OWASP Top 10 체크리스트 준수
✅ 감사 로깅 구현

추가 계획:
□ 정기 보안 감사 (분기별)
□ 침투 테스트 (PMF 달성 후)
□ 버그 바운티 프로그램 (스케일업 시)

비용: 외부 감사 $5,000/분기

- 24시간 내 긴급 리뷰

- 외부 자문 검토

- 모든 작업 중단
- 비상 대응 계획 실행
- 이해관계자 즉시 통보

## 8. 비상 대응 계획

### 8.1 기술 장애 대응

```
Supabase/Vercel 장애 발생 시:

1. 상태 확인 (5분)
   - status.supabase.com
   - status.vercel.com
   - 내부 로그 확인

2. 사용자 커뮤니케이션 (10분)
   - 서비스 상태 페이지 업데이트
   - 예상 복구 시간 안내

3. 복구 작업
   - 캐시 데이터 활용
   - 읽기 전용 모드 전환
   - 백업 리전 활성화 (가능 시)

4. 사후 분석
   - 인시던트 리포트 작성
   - 재발 방지책 수립
```

### 8.2 번아웃 대응

```
번아웃 경고 신호 감지 시:

1단계 (경고):
- 주간 업무 시간 감축 목표 설정
- 비필수 작업 연기
- 다음 주 "결정 없는 날" 2일로 확대

2단계 (심각):
- 1주일 강제 휴식
- 핵심 운영만 유지 (자동화 의존)
- 멘토/지인에게 SOS

3단계 (위기):
- 2주 이상 휴식
- 서비스 최소 운영 모드
- 전문가 상담 고려
```
