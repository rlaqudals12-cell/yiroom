# /ux-check 명령어

변경된 UI 컴포넌트/페이지에 대해 UX 체크리스트를 자동 점검합니다.

## 대상

$ARGUMENTS (미지정 시 최근 git 변경 파일 중 UI 관련 파일)

## 실행 절차

### 1. 대상 파일 식별 + 화면 유형 추론

- 인자가 있으면: 해당 파일/화면 관련 컴포넌트 탐색
- 인자가 없으면: `git diff --name-only HEAD` 에서 `.tsx` 파일 추출
- 페이지(`app/`)와 컴포넌트(`components/`) 모두 포함
- UI 관련 파일이 없으면: "점검 대상 UI 파일이 없습니다" 출력 후 종료

파일 경로에서 **화면 유형**을 추론하여 체크리스트의 "화면 유형별 프로필" 적용:

- `page.tsx` (루트), `LandingContent` → 랜딩 페이지 프로필
- `home`, `dashboard`, `(tabs)/index` → 홈/대시보드 프로필
- `analysis/*/result` → 분석 결과 프로필
- `analysis/*` (result 제외) → 분석 입력 프로필
- `settings`, `profile` → 설정 프로필
- `capsule` → Daily Capsule 프로필
- 매칭 불가 시 → 전체 항목 점검 (프로필 미적용)

### 2. 체크리스트 로드 + 자동 감지 + 판정

`docs/checklists/ux-pr-checklist.md` 파일을 읽는다.
각 항목의 **"판정 기준"**과 **"탐지 방법"** 컬럼을 기준으로 대상 코드를 분석한다.

> 중요: 탐지 로직은 체크리스트 파일이 Single Source of Truth.
> 이 명령어 파일에 탐지 패턴을 중복 정의하지 않는다.

#### 자동 감지 (Auto-Detection)

기본 A~G 점검에 더해, 대상 코드의 패턴을 분석하여 **H~K 카테고리를 자동 활성화**한다.
별도 플래그 없이 `/ux-check` 한 번으로 모든 해당 카테고리가 적용된다.

| 감지 조건                                              | 활성화 항목                       | 점검 내용                                             |
| ------------------------------------------------------ | --------------------------------- | ----------------------------------------------------- |
| `Tabs`, `TabsList`, `TabsTrigger` 컴포넌트 감지        | **H1~H3** (탭 설계)               | 콘텐츠 충실도, 기능 밀도 균형, 탭 간 정보량 균형      |
| 조건부 렌더링 (`{condition && <C/>}`, 삼항) 감지       | **H4** (fallback 품질)            | 조건 미충족 시 유용한 대안 존재 여부 — 탭 유무와 무관 |
| `<details>`, 아코디언 패턴 감지                        | **H5** (접힘 발견성)              | summary 텍스트와 시각적 구분 — 탭 유무와 무관         |
| `analysis/*/result` 경로의 result 페이지 2개 이상 존재 | **I1~I4** (교차 모듈 일관성)      | 공통 컴포넌트, 용어 통일, 결과 구조, 공유 패턴        |
| `dark:` Tailwind 클래스 사용                           | **J1~J3** (다크/라이트 모드)      | 전체 적용, 대비 유지, 이미지/아이콘                   |
| 분석 결과 화면 (analysis/\*/result)                    | **K1~K2** (신뢰/투명성 시그널)    | 신뢰도 표시, 분석 한계 안내                           |
| `ProgressiveProfilePrompt` import 감지                 | **L1~L4** (Progressive Profiling) | 타이밍, 비강제성, 가치 교환, 피드백                   |
| `AnalysisMatchedProducts`/`ProductCard`+`matchScore`   | **M1~M5** (제품 추천/수익 UX)     | 자연스러운 연결, 투명성, 빈 상태, 비침습, 캐싱        |
| `EnvironmentAdviceCard`/`getCurrentWeather` 감지       | **N1~N4** (환경/실시간 데이터)    | Graceful degradation, 실행 가능성, 캐시, 과부하       |

**감지 로직 상세:**

- **H1~H3 (탭 설계)**: 대상 파일 또는 import 체인에서 `Tabs`, `TabsList`, `TabsTrigger`, `TabsContent` 사용 여부 확인. 감지 시 각 탭의 콘텐츠 충실도와 기능 밀도 균형을 점검.
- **H4 (fallback 품질)**: 탭 유무와 무관. 대상 파일에 `{condition && <Component>}` 또는 삼항 조건부 렌더링이 있으면 활성화. 조건 미충족 시 빈 화면이 아닌 유용한 대안 UI가 존재하는지 점검.
- **H5 (접힘 발견성)**: 탭 유무와 무관. `<details>`/`<summary>` 또는 아코디언 패턴이 감지되면 활성화. 접힌 콘텐츠의 존재를 사용자가 인지할 수 있는지 점검.
- **I1~I4 (교차 모듈)**: `app/(main)/analysis/` 하위에 `result/*/page.tsx` 패턴의 result 페이지가 2개 이상 존재하면 활성화. 공통 컴포넌트(ConsultantCTA, AITransparencyNotice, ShareButton 등) 사용 일관성 확인.
- **J1~J3 (다크 모드)**: 대상 파일에 `dark:` 접두사 클래스가 1개 이상 있으면 활성화. 전체 적용 여부와 대비율 유지 확인.
- **K1~K2 (신뢰 시그널)**: 화면 유형이 "분석 결과"이면 자동 활성화. AI 분석 신뢰도 표시, 분석 한계 안내 존재 여부 점검.
- **L1~L4 (Progressive Profiling)**: `ProgressiveProfilePrompt` import 감지 시 활성화. 결과 하단 위치 확인(L1), 접이식 기본(L2), accuracyBoost 표시(L3), 제출 후 피드백(L4).
- **M1~M5 (제품 추천)**: `AnalysisMatchedProducts` 또는 `ProductCard`+`matchScore` 감지 시 활성화. 분석-제품 순서(M1), matchScore 렌더링(M2), 빈 상태 UI(M3), 핵심 결과 비침습(M4), 캐시 TTL(M5).
- **N1~N4 (환경 데이터)**: `EnvironmentAdviceCard` 또는 `getCurrentWeather` 감지 시 활성화. API 실패 시 null(N1), 구체적 행동 조언(N2), 캐시 30분 TTL(N3), topAdvice ≤ 5개(N4).

판정 결과:

- **pass**: 코드에서 준수가 확인됨
- **fail**: 위반 발견 (구체적 위치 명시)
- **warn**: 자동 판정 불확실, 수동 확인 필요 (체크리스트에서 "warn 고정" 표시된 항목)
- **n/a**: 해당 없음 (예: 비동기 로드 없는 컴포넌트의 C1 로딩 상태)

### 3. 모드

#### 기본 모드 (전체)

A~N 전 카테고리 70항목 점검 (화면 유형 프로필 + 자동 감지에 의해 n/a 자동 스킵).

#### --quick 모드

Critical/High 심각도만 빠르게 점검:

- **A1, A3** (영어 raw value, err.message) — Critical
- **D1~D4** (감정 설계 금지 패턴) — Critical
- **A2, B1, B2** (해요체, data-testid, 터치 44px) — High
- **B6** (Safe Area — 모바일 대상 시) — High
- **C1~C3** (3종 상태) — High
- **D6** (프라이버시 안내 — 카메라/업로드 대상 시) — High
- **D9** (자율성 존중 — 지시형 표현) — High
- **D10** (비의료 고지 — 피부/구강/영양 결과 시) — High
- **H1, H2** (탭 콘텐츠 충실도, 기능 밀도 균형 — 탭 감지 시) — High
- **I4** (공유 패턴 적용 — 교차 모듈 감지 시) — High
- **J1** (다크 모드 전체 적용 — dark: 감지 시) — High
- **L1, L2** (Progressive Profiling 타이밍/비강제성 — 감지 시) — High
- **M1, M3, M4** (제품 추천 자연스러움/빈 상태/비침습 — 감지 시) — High
- **N1** (환경 데이터 graceful degradation — 감지 시) — High

= 14~26항목 (자동 감지 결과에 따라 가변), 1분 내 완료.

### 4. 사용자 관점 시뮬레이션 (기본 모드에서 자동 실행)

코드 분석 후, 대상 화면에 대해 시뮬레이션을 수행한다.

#### 4a. 1초 테스트 (Glance Test — Jobs T3)

> "1초 만에 이 화면의 목적이 보이는가?"

화면의 JSX 구조에서 시각적 위계를 분석:

- **1초 후 기억에 남는 것**(가장 큰 텍스트/아이콘/색상 요소)이 핵심 가치와 일치하는가?
- 시선 흐름이 **핵심 → CTA** 순서로 자연스러운가?
- 장식 요소가 핵심을 가리지 않는가?

#### 4b. 페르소나 시뮬레이션

화면 유형에 따라 **가장 중요한 페르소나**를 자동 선택하여 시뮬레이션:

| 화면 유형   | 1순위 페르소나 | 2순위         | 핵심 질문                                             |
| ----------- | -------------- | ------------- | ----------------------------------------------------- |
| 랜딩        | **P1 탐색자**  | P5 인플루언서 | "3초 안에 뭐 하는 앱인지 알겠어?" / "콘텐츠 소재됨?"  |
| 홈/대시보드 | **P3 일상화**  | P1 탐색자     | "오늘 새로운 게 있나?" / "뭘 먼저 해야 해?"           |
| 분석 결과   | **P2 학습자**  | P4 복귀자     | "이게 나한테 무슨 의미지?" / "예전 결과랑 뭐가 달라?" |
| 분석 입력   | **P1 탐색자**  | —             | "어떻게 시작하지? 어려울까?"                          |
| 캡슐        | **P3 일상화**  | P4 복귀자     | "오늘 나한테 맞는 조언이 있나?"                       |
| 설정        | **P2 학습자**  | —             | "원하는 설정을 바로 찾을 수 있나?"                    |

**페르소나 정의**:

- **P1 탐색자**: 처음 방문, "이게 뭔데?"
- **P2 학습자**: 분석 결과 확인 중, "이해하고 싶다"
- **P3 일상화**: 매일 사용, "오늘 뭐가 달라?"
- **P4 복귀자**: 한동안 안 씀, "뭐가 바뀌었지?"
- **P5 인플루언서**: 콘텐츠 소재 탐색, "이거 캡처해서 올릴 수 있나?"

각 페르소나 관점으로:

- **3초 진입**: "이 화면의 핵심 가치를 이해할 수 있는가?"
- **CTA 명확성**: "다음 행동이 명확한가?"
- **Delight 존재**: "이 경험에 '와!' 하는 순간이 있는가?" (Jobs T6)
- 문제 발견 시 점검 결과 하단에 "사용자 시뮬레이션 소견" 추가

#### 4c. 탭별 시뮬레이션 (H 카테고리 활성화 시)

탭이 감지된 화면에서 추가 시뮬레이션:

- 각 탭을 독립 페이지처럼 평가: "이 탭만 봐도 가치가 있는가?"
- 탭 간 콘텐츠 밀도 편차: "특정 탭이 비어 보이지 않는가?"
- 탭 전환 동기: "사용자가 다른 탭을 클릭할 이유가 명확한가?"

### 5. 궁극의 형태 체크 (해당 시)

대상이 주요 화면(랜딩, 홈, 분석 결과, 대시보드)인 경우:

- `docs/TODO.md` 섹션 11 "모듈별 궁극의 형태 (100점 정의)" 참조
- "이 화면이 궁극의 형태 대비 몇 %인가?" 소견 추가
- 부족한 부분이 의도적 제외인지 누락인지 판단
- 성능 최적화 상태 확인 (dynamic import, 캐싱 적용 여부)

### 6. 실패 패턴 대조 (자동)

점검 결과의 fail 항목을 `docs/checklists/ux-known-issues.md`의 기존 패턴과 대조한다.

- 기존 패턴과 일치하면: "기존 패턴 KI-00X 재발" 경고 + 발견 횟수 증가
- 새로운 유형의 fail이면: "신규 패턴 후보" 표시
- 대조 후 known-issues 파일의 발견 횟수를 업데이트한다

### 7. 이력 저장 (--save 모드)

`--save` 플래그 사용 시 점검 결과를 파일로 저장:

- 저장 위치: `docs/checklists/history/YYYY-MM-DD-{화면유형}.md`
- 이전 이력이 존재하면 통과율 비교 표시 ("18/32 → 24/32, +6 개선")
- 궁극의 형태 % 트렌드 추적

## 사용 예시

```
/ux-check                          # 최근 변경 전체 점검 (자동 감지 포함)
/ux-check 홈 화면                  # 특정 화면 지정 (홈 프로필 적용)
/ux-check components/home/*.tsx    # 파일 지정
/ux-check --quick                  # Critical+High만 빠르게
/ux-check --save                   # 결과 이력 저장
/ux-check 퍼스널컬러 분석 결과     # 분석 결과 프로필 + K 자동 활성화
/ux-check 홈 화면 --quick --save   # 조합 사용
```
