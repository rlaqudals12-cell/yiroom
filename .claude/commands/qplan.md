# /qplan 명령어

빠른 계획 분석 및 검토를 수행합니다.

## 연동 에이전트

- **yiroom-spec-reviewer**: 스펙 논리적 허점 검토
- **Explore**: 코드베이스 탐색

---

## 분석 워크플로우

```mermaid
graph TD
    A[/qplan 호출] --> B[요구사항 분석]
    B --> C[코드베이스 탐색]
    C --> D[유사 구현 확인]
    D --> E{기존 패턴 존재?}
    E -->|Yes| F[패턴 재사용 계획]
    E -->|No| G[새 패턴 설계]
    F --> H[리스크 분석]
    G --> H
    H --> I[계획 출력]
    I --> J[승인 요청]
```

---

## 실행 내용

$ARGUMENTS에 대해:

### 1. 요구사항 분석 (P0 적용)

| 질문           | 목적           |
| -------------- | -------------- |
| 왜 필요한가?   | 요구사항 의심  |
| 삭제하면?      | 최소 범위 확인 |
| 누가 요청했나? | 근거 추적      |

### 2. 코드베이스 분석

**탐색 대상**:

- 유사한 기존 구현 (Grep/Glob)
- 재사용 가능한 컴포넌트
- 공통 유틸리티 함수
- 기존 타입 정의

### 3. 일관성 검증

| 체크 항목 | 기준                    |
| --------- | ----------------------- |
| 네이밍    | 기존 컨벤션 준수        |
| 파일 위치 | 기존 구조 일치          |
| 패턴      | Repository/Service 패턴 |
| 테스트    | 기존 테스트 구조        |

### 4. 리스크 식별

```markdown
### ⚠️ 리스크 분석

| 리스크    | 영향도         | 완화 방안 |
| --------- | -------------- | --------- |
| [리스크1] | 높음/중간/낮음 | [방안]    |
```

### 5. 의존성 확인

- **선행 조건**: 필요한 사전 작업
- **영향 범위**: 변경 시 영향받는 파일
- **테스트 범위**: 검증 필요 영역

---

## 출력 형식

```markdown
## 📋 구현 계획

### 요구사항 분석

- **목적**: [왜 필요한지]
- **범위**: [무엇을 변경하는지]

### 재사용 가능 코드

| 파일       | 용도   | 재사용 방법 |
| ---------- | ------ | ----------- |
| [파일경로] | [설명] | [방법]      |

### 변경 파일

| 파일   | 변경 유형 | 설명   |
| ------ | --------- | ------ |
| [파일] | 신규/수정 | [설명] |

### 리스크

- [식별된 리스크 및 완화 방안]

### 예상 작업량

- **소요 시간**: ~X분
- **복잡도**: 낮음/중간/높음
- **테스트**: [필요 테스트]

### 진행하시겠습니까? (Y/N)
```

---

## 복잡도 기준

| 복잡도   | 기준                   | 권장              |
| -------- | ---------------------- | ----------------- |
| **낮음** | 1-2 파일, 검증된 패턴  | 직접 구현         |
| **중간** | 3-4 파일, 일부 새 패턴 | /qcode 후 /qcheck |
| **높음** | 5+ 파일, DB/인증 관련  | /sisyphus 권장    |

---

## 사용 예시

```bash
# 기본 사용
/qplan 로그인 버튼 추가

# 페이지 구현
/qplan PC-1 결과 페이지 구현

# API 추가
/qplan 이미지 업로드 API

# 버그 분석
/qplan 429 에러 원인 분석
```
