# 문서 동기화 규칙 (Document Sync Rule)

> 코드 변경 시 관련 문서도 함께 업데이트하는 필수 체크리스트
> **P7 보강**: 구현 후에도 문서-코드 동기화를 유지

---

## 핵심 원칙

```
코드와 문서는 항상 동기화되어야 한다.
코드만 변경하고 문서를 업데이트하지 않으면 → 기술부채.
문서만 있고 코드가 없으면 → 죽은 문서.
```

---

## 변경 유형별 문서 업데이트 매트릭스

### 1. DB 스키마 변경

| 변경              | 필수 업데이트 문서                                           |
| ----------------- | ------------------------------------------------------------ |
| 새 테이블 추가    | `docs/DATABASE-SCHEMA.md` (테이블 목록 + 상세 섹션 + 관계도) |
| 컬럼 추가/수정    | `docs/DATABASE-SCHEMA.md` (해당 테이블 섹션)                 |
| RLS 정책 변경     | `docs/DATABASE-SCHEMA.md` + 해당 ADR                         |
| 마이그레이션 추가 | 배포 체크리스트 (`docs/troubleshooting/`)                    |

### 2. 새 기능/모듈 추가

| 변경              | 필수 업데이트 문서                                                                                                                              |
| ----------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| 새 도메인 모듈    | ADR (`docs/adr/` + `docs/adr/README.md` 인덱스) + 원리 (`docs/principles/` + `README.md`) + 스펙 (`docs/specs/`) + `CLAUDE.md` 모듈 현황 테이블 |
| 새 API 엔드포인트 | 해당 스펙 문서 + ADR (아키텍처 결정이 있으면)                                                                                                   |
| 새 컴포넌트 패턴  | 해당 ADR (새 패턴 도입 시)                                                                                                                      |
| 타입 변경         | 해당 스펙 문서의 타입 정의 섹션                                                                                                                 |

### 3. 아키텍처 변경

| 변경             | 필수 업데이트 문서                                                  |
| ---------------- | ------------------------------------------------------------------- |
| 기술 선택 변경   | ADR 신규 작성 (대안 + trade-off 기록) + `docs/adr/README.md` 인덱스 |
| 모듈 구조 변경   | `docs/ARCHITECTURE.md` + 해당 모듈 BOUNDARIES.md                    |
| 의존성 방향 변경 | `encapsulation.md` 규칙 위반 여부 확인                              |

### 4. 버그 수정 / 리팩토링

| 변경               | 필수 업데이트 문서              |
| ------------------ | ------------------------------- |
| 단순 버그 수정     | 없음 (코드 변경만)              |
| 동작 변경하는 수정 | 해당 스펙 문서 (입출력 변경 시) |
| 대규모 리팩토링    | ADR (결정 근거 기록)            |

### 5. 규칙/에이전트 변경

| 변경          | 필수 업데이트 문서                                     |
| ------------- | ------------------------------------------------------ |
| 새 규칙 파일  | `.claude/rules/README.md` (인덱스 테이블 + P원칙 매핑) |
| 에이전트 변경 | `.claude/agents/README.md`                             |

## 자동 체크 질문 (커밋 전)

코드 변경 완료 후, 커밋 전에 다음 질문에 답한다:

```
□ 1. DB 스키마 변경이 있는가?
     → DATABASE-SCHEMA.md 업데이트했는가?
     → 배포 체크리스트에 마이그레이션 추가했는가?

□ 2. 새 기능/모듈인가? ★ ADR 필수 체크
     → ADR이 존재하는가? (없으면 반드시 작성)
     → ADR README 인덱스 목록에 추가했는가?
     → ADR README 카테고리 섹션에 추가했는가?
     → 원리 문서가 존재하는가? (도메인 지식이 있는 기능은 필수)
     → 원리 README 인덱스에 추가했는가?
     → 스펙 문서가 존재하는가? (입출력 정의가 있는 기능은 필수)
     → 스펙 README 인덱스에 추가했는가?
     → CLAUDE.md 모듈 현황 테이블을 업데이트했는가?

□ 3. 기존 모듈의 확장 기능인가? ★ ADR 필수 체크
     → 기존 ADR에 추가할 수 있는가? (없으면 새 ADR 작성)
     → 기존 스펙에 추가할 수 있는가? (없으면 새 SDD 작성)
     → 기존 원리 문서로 커버되는가? (안 되면 새 원리 문서 작성)
     → 모든 인덱스(ADR/스펙/원리 README) 업데이트했는가?

□ 4. 아키텍처 결정을 내렸는가?
     → ADR에 결정 + 대안 + trade-off가 기록되었는가?
     → ADR README 인덱스 + 카테고리에 추가했는가?

□ 5. 기존 문서의 내용이 변경된 코드와 불일치하는가?
     → 해당 문서를 업데이트했는가?

□ 6. 규칙/에이전트 파일을 추가/수정했는가?
     → README 인덱스를 업데이트했는가?
```

### ADR/스펙/원리 동기화 의무 규칙

```
새 lib/ 모듈 또는 기존 모듈 확장 시:

필수:
  - ADR 작성 (또는 기존 ADR 참조 가능 확인)
  - ADR README 인덱스 목록 + 카테고리 섹션 추가
  - 스펙(SDD) 작성 (입출력이 있는 기능)
  - 스펙 README 인덱스 추가

권장 (도메인 지식 기반):
  - 원리 문서 작성 (과학적/학문적 근거가 있는 기능)
  - 원리 README 인덱스 추가

예외:
  - 단순 유틸리티 함수 (formatDate 등) → ADR/스펙 불필요
  - 기존 ADR로 완전히 커버되는 변경 → 새 ADR 불필요 (기존 ADR 참조)
```

## 문서 위치 요약

```
docs/
├── DATABASE-SCHEMA.md          # DB 스키마 (테이블 추가/변경 시)
├── ARCHITECTURE.md             # 아키텍처 개요 (구조 변경 시)
├── adr/                        # Architecture Decision Records
│   ├── README.md               # ADR 인덱스 (새 ADR 시 목록 추가)
│   └── ADR-NNN-*.md            # 개별 ADR
├── principles/                 # 도메인 원리 (새 도메인 시)
│   ├── README.md               # 원리 인덱스
│   └── *.md                    # 개별 원리 문서
├── specs/                      # 스펙 문서 (새 기능 시)
└── troubleshooting/            # 배포/운영 (마이그레이션 시)
```

## 위반 감지

### 코드 리뷰 시 확인

1. **새 마이그레이션 파일** (`supabase/migrations/`) → DATABASE-SCHEMA.md 변경 있는가?
2. **새 ADR 번호 파일** (`docs/adr/ADR-*.md`) → README.md 인덱스(목록+카테고리)에 추가됐는가?
3. **새 lib/ 모듈** → 관련 ADR + 스펙(SDD)이 존재하는가? (없으면 작성 필수)
4. **새 스펙 파일** (`docs/specs/SDD-*.md`) → README.md 인덱스에 추가됐는가?
5. **새 원리 파일** (`docs/principles/*.md`) → README.md 인덱스에 추가됐는가?
6. **타입 변경** (`types.ts`) → 스펙 문서와 일치하는가?

### 커밋 메시지 힌트

문서 업데이트가 포함된 커밋은 scope에 `docs`를 포함:

```
feat(web): ConnectionAwareness 내재화 추적 시스템
docs: ADR-083 ConnectionAwareness 아키텍처 + DATABASE-SCHEMA v7.1
```
